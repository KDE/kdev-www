从哪里可以得到KDevelop？ 怎么安装？
KDevelop没有发布任何二进制版本的KDevelop (RPMs). RPMs可以在发行商的服务器上或在下面相应的子目录中找到 ftp.kde.org 或它的镜像
请看看 我们的下载站点 for 获得如何下载和编译源代码的更多信息


我可以用KDevelop开发商用程序吗?
任何个人和组织可以自由地使用KDevelop。生成的代码可以使用您所希望的任何许可证。所以项目生成器有一个加入您的版权通告的域，使其用于生成的文件；缺省的GPL许可证通告只是是为了提供实例，同时简化您的工作。所以您可以在此加入您的版权通告。
当编辑您的程序的时候，使用KDevelop和使用其他任何编辑器一样，它只是一个使用了GPL的程序，对于您使用它的目的，没有任何的暗示。所以您可以开发商用/封闭代码的软件，尽管KDevelop本身是GPL的。
对于使用KDE功能和其他自由软件，您只需要注意您使用的库的许可证。 所以开发商用的KDE/Qt应用，您所需要的只是从Trolltech获取专业版许可证。 KDE库是LGPL的，这意味着您可以使用函数调用，并把您的程序连接到KDE库。这包括了kdelibs库，kdebase库(konqueror和kcontrol等)和koffice库(用于创建koffice 应用)。
更多信息，请访问 KDevelop 许可证


我可以为KDevelop开发商用的插件吗?
KDevelop包含了libkdeveloplgpl.so用于开发商用的插件。这个库包括了KDevelop 界面和实用工具，并且采用LGPL发布。如果从Trolltech获取了专业版许可证，你可以把您的代码连接到libkdeveloplgpl，以任何许可证方式发表。


我编译和安装了KDevelop 3.0，但是我看到的只有File和Help菜单
和所有其他的KDE应用一样，您必须保证KDevelop的插件可以被 KDE感知。您可以把它安装到您的KDE目录(比如configure --prefix=/opt/kde3 ) 或者把KDevelop路径加入到KDEDIRS环境变量中，并再次运行"kbuildsycoca"。 bash命令行的例子(假设 KDevelop安装在/home/harry/kdevelop):
export KDEDIRS=/home/harry/kdevelop:$KDEDIRS && kbuildsycoca


我怎样通过键盘来切换打开的文件？
您可以使用ALT-Left和ALT-Right来切换打开的文件。这些键可以在KDevelop setup中配置。还有一个Quick Open (CTRL-SHIFT-O) 来选中和打开项目中的文件；Switch To (CTRL-/) 允许您输入打开的文件的头几个字母进行切换。


在调试控制台程序的时候，我怎样才能输入文字？
GDB允许您输入自己的命令。


对于外部库，如何打开代码补齐？
当您打开任何C/C++项目的时候，进入菜单Project->Project Settings, 在 Project Options 对话框中选择C++ Specific tab, 然后选择 Code Completion 标签。 点击 Add Persistant Class Store 按钮,完成New Persistant Class Store向导。 最后，确定在Code Completion 标签页中的永久类存储列表，选取新产生的类存储。

我在开发应用的时候，需要定义在预编译头中定义 -D选项 如果我把它加入到KDevelop的项目选项中，没有任何问题。但是当我发布我的包的时候，-D选项没有使用，我还要做些什么？
编辑最顶层源文件目录下的configure.in(.in)文件。 在AC_CHECK_COMPILERS宏以后加入：
CPPFLAGS="$CPPFLAGS -DMYDEFINE"
然后您必须通过
"make -f Makefile.dist" 重新生成"configure"然后回到configure脚本。


有其他方式来发布我的定义吗？
另外一个在您的项目中包含DEFINE的方法是使用config.h.
您必须为下列文件打补丁：
在项目所在的最高层目录的acconfig.h加入下列行：
#undef MYDEFINE

然后在configure.in(.in)文件中，在宏
AC_CONFIG_HEADER(config.h)之后加入：
AC_DEFINE_UNQUOTED(MYDEFINE)

(当您运行下列命令来更新项目，它就会在config.h中定义
"make -f Makefile.dist; configure [your options]".

这个解决方案的条件是：
AC_CONFIG_HEADER(config.h)
在configure.in(.in)和源代码中所有的其他包含文件之前有下列块
#ifdef HAVE_CONFIG_H
#include <config.h>
#endif
(通常我们的项目模板已经作了相应设置)。

此外：这种方案的另外一个优点是使得MYDEFINE仅依赖于
特定的条件（为configure设置一个选项）。
这适合于写一个您自己的宏函数。


我在开发的一个程序需要异常处理，我应该怎么做？
编辑最顶层源文件目录下的configure.in(.in)文件。 在AC_CHECK_COMPILERS宏以后加入：
CXXFLAGS="$CXXFLAGS $USE_EXCEPTIONS"
然后您必须在最顶层源文件目录下通过
"make -f Makefile.dist" 重新生成"configure"， 然后重新启动configure脚本。


我的configure.in.in没有任何AC_CHECK_COMPILERS()调用。我应该在哪里放置上面提到的 CXXFLAGS="$CXXFLAGS $USE_EXCEPTIONS" ?
新版本的configure.in.in已经作了简化，您可以在
AM_INIT_AUTOMAKE(<package-name>,<version>)
之后加入该行。


我开发了一个应用程序，需要使用运行时类型信息，
我应该怎么做？
编辑最顶层源文件目录下的configure.in(.in)文件。 在AC_CHECK_COMPILERS宏以后加入：
CXXFLAGS="$CXXFLAGS $USE_RTTI"
然后您必须在最顶层源文件目录下通过
"make -f Makefile.dist" 重新生成"configure"， 然后重新启动configure脚本。



上述的方案很好，但是我仅需要运用到某些子目录（比如项目中的静态库），
我应该怎么做？
编辑您希望的源代码目录中的Makefile.am文件。这里，您
需要在KDevelop特定部分以外的什么地方输入：
KDE_CXXFLAGS=$(USE_EXCEPTIONS)
请注意，只有您的项目使用am_edit才会奏效。
(它需要较新版本的acinclude.m4 & am_edit, AFAWK 它们必须
发布于12-09-1999之后，以前的版本被称为APPEND_CXXFLAGS -
在KDevelop的发行版中，您可以找到一个合适的版本。)

您需要如上所述更新Makefile
(make -f Makefile.dist; ./configure [选项]).


CPPFLAGS和CXXFLAGS有什么区别？
CPPFLAGS用于定义(即这个标志仅仅用于编译)
CXXFLAGS用于编译器开关(比如 -frtti),它同时作用于
链接过程。


我看到一些方案使用给Makefile.am打补丁的方法，
DEFS+=-DMYDEFINE
或
CXXFLAGS+=-fsigned-char
或
LD_FLAGS+=-L/usr/local/lib
为什么你们不考虑这样？
在automake 1.4存在着一个缺陷，它不能为已经存在的标志加入新的标志。
在Makefile.am中使用以上命令，您会删除现存的标志。
如果您只希望将这些标志应用于项目的子目录：
AM_CPPFLAGS=-DMYDEFINE
或
AM_CXXFLAGS=-fsigned-char
或
AM_LDFLAGS=-L/usr/local/lib

请注意这些标志只有在automake1.4版本以上才会奏效。

要为整个项目设置标志，请为最高级源代码目录下的
configure.in(.in)文件打补丁。


我怎么保证我的项目的用户使用automake 1.4来编译?
在最顶层源文件目录下的Makefile.am文件中加入AUTOMAKE_OPTIONS变量，其值为字符串"1.4"。比如：
AUTOMAKE_OPTIONS = foreign 1.4


我在项目中加入了一个现存的子项目，但是它并没有编译。
如果builddir != topsrcdir，会发生这种情况。您可以在加入子项目之前运行make distclean。


AM_CXXFLAGS (在Makefile.am中) 可以使用-frtti 或
-fexceptions这样的标志吗？
不行！这种情况下存在一个顺序的问题。标准的CXXFLAGS变量
是通过acinclude.m4输出的，它已经包含了-fno-rtti (-fno_exceptions)。
如果您将它加入AM_CXXFLAGS编译器将会把这些标志在
CXXFLAGS之前插入。比如 -frtti:
gpp ... -frtti ... -fno-exceptions -fno-rtti ......
所以最后的标志(本例中，-fno-rtti)将会起作用。
KDE_CXXFLAGS会在标准的CXXFLAGS之后插入，所以它就会是这样：
gpp ..... -fno-exceptions -fno-rtti ...... -frtti...
它可以很好的工作。

请注意： KDE_CXXFLAGS只能通过项目发布，它使用am_edit
和acinclude.m4.


为什么打了CXXFLAGS="$USE_EXCEPTIONS $CXXFLAGS"补丁的configure.in(.in)没有用？
请参见 "AM_CXXFLAGS (在Makefile.am中)可以用-frtti或 -fexceptions这样的标志吗? "问题的答案 ;-)


我在configure.in(.in)中加入CXXFLAGS="$USE_EXCEPTIONS"，一切都工作正常，
为什么我需要用你们建议的方案？
如果您是在AC_CHECK_COMPILERS()之前这样做，
CXXFLAGS="-pedantic" ./configure
将不会正常的工作。
如果您是在AC_CHECK_COMPILERS()之后这样做，在
AC_CHECK_COMPILERS()中设置的 CXXFLAGS将会被清除。
(比如"configure --enable-debug"，CXXFLAGS将会
把调试信息加入到您的代码中)。
所以请用 CXXFLAGS="$CXXFLAGS $USE_EXCEPTIONS" 。


我在我的KDE/QT项目中使用了额外的头文件和库，将
-I/usr/include/foo加入编译选项，-L/usr/lib/foo加入链接选项，并没有把这个配置加入tar包。
我应该怎么做？
最难的部分就在于如何正确处理的这个问题。
一个简单，但是并非正确的方案是在configure.in(.in)加入：

all_includes="$all_includes -I/usr/include/foo"
all_libraries="$all_libraries -L/usr/lib/foo"

[最好的加入上述语句的位置是在文件的末尾，:
AC_ARG_PROGRAM
AC_OUTPUT( ..... )
语句之前]

存在的问题：
- 谁能够保证所有的包含文件总是在
/usr/include/foo...也许在某些系统中，它在/usr/local/foo/include
(也许库在 /usr/local/foo/lib)??
- 也许我没有安装foo之类的东西，编译失败了，但是也许
configure失败，并提供有用的错误信息会更好。

所以，您也许需要在configure.in(.in)中加入一个宏函数，
它将作一些测试，并搜索是否安装了"foo"。

如果您要这样做的话，我们在这里给您一些提示：
试图理解acinclude.m4.in或者产生一个KDE-normal-ogl项目，
这次您会找到一个qgl.m4文件，他将和acinclude.m4连接在一起(请参考 "Makefile.dist")。
它是一个用来搜索QT-OpenGL-Lib的很好的例子
您可以在此基础上重写来解决您的问题。

如果您的运气好的话，您可以在acinclude.m4.in中找到问题的解决方案，
这样您就可以用已经产生的函数。


上述问题中，我总是看到"configure.in(.in)"。在我的项目中，两个文件都有，
哪一个文件应该打补丁？
如果在最顶层源文件目录下有configure.in.in，就为这个文件打补丁。
如果没有configure.in.in，就用configure.in。

我已经安装了htdig，但是htsearch不能用？
