Où puis-je me procurer KDevelop ? Comment l'installer ?
L'équipe de KDevelop ne distribue aucune version binaire de KDevelop (RPMs). Les RPMs se trouvent toutefois sur le serveur de votre distributeur ou dans les sous-répertoires correspondant de ftp.kde.org ou sur un de ses mirroirs
Visitez notre site de téléchargement pour des informations concernant la façon de télécharger et de compiler le code source.


Suis-je autorisé à développer des applications commerciales avec KDevelop ?
Tout le monde et toutes les entités sont libres d'utiliser KDevelop. Le code généré peut être sous toutes les licences que vous voulez. Toutefois, le générateur de projet possède les champs nécessaires pour ajouter vos propres notes de copyright aux fichiers générés, mais pour simplifier les choses et pour donner un exemple, par défaut, c'est la notice de la license GPL. Donc, vous pouvez ajouter votre licence là.
Quand c'est pour éditer votre programme, utiliser KDevelop, c'est simplement comme utiliser n'importe quel autre éditeur, c'est un programme sous GPL et ça n'a pas d'implications sur le but dans lequel vous l'utilisez. Donc vous pouvez écrire des logiciels commerciaux/fermés malgré que KDevelop soit lui-même sous GPL.
En ce qui concerne l'utilisation des fonctions KDE ou tout autre chose que du logiciel libre, vous devez seulement regarder les notes de licence sur les bibliothèques que vous allez utiliser. Donc pour le développement commercial d'applications KDE/Qt, vous avez uniquement besoin d'une licence professionnelle de Trolltech. Les bibliothèques KDE elles-mêmes sont sous LGPL, ce qui veut dire que vous pouvez utiliser les appels de fonctions et lier votre programme aux bibliothèques KDE. Ca vaut pour les bibliothèques kdelibs, les bibliothèques kdebase (konqueror et kcontrol etc.) et les bibliothèques koffice (pour créer des applications koffice).
Pour plus d'informations, visitez les licences KDevelop


Suis-je autorisé à développer des plugins commerciaux pour KDevelop ?
KDevelop inclut une bibliothèque libkdeveloplgpl.so pour permettre le développement de plugins commerciaux Cette bibliothèque inclut des interfaces et des fonctions utilitaires de KDevelop sous LGPL. Si vous avez une licence professionnelle de Trolltech vous pouvez lier votre code avec libkdeveloplgpl et le distribuer sous n'importe quelle licence.


J'ai compilé et installé KDevelop 3.0 mais tout ce que je vois, ce sont les menus Fichier et Aide
Juste comme n'importe quelle autre application KDE, vous devez vous assurer que les plugins KDevelop sont reconnus par KDE. Installez-les dans votre répertoire KDE (configure --prefix=/opt/kde3 par exemple) ou ajouter le chemin de KDevelop dans la variable d'environnement KDEDIRS et relancez "kbuildsycoca". Exemple de ligne de commande pour bash (en supposant que KDevelop est installé dans /home/harry/kdevelop) :
export KDEDIRS=/home/harry/kdevelop:$KDEDIRS && kbuildsycoca


Comment passer d'un fichier ouvert à un autre avec le clavier ?
Vous pouvez utiliser ALT-Left et ALT-Right pour aller au fichier ouvert précédent/suivant. Ces touches peuvent être configurées dans la configuration de KDevelop. Il y a aussi une fonction Ouverture Rapide (CTRL-SHIFT-O) pour sélectionner et ouvrir un fichier dans un projet et une fonction Aller A (CTRL-/) qui permet de taper les premières lettres du nom d'un fichier ouvert pour s'y rendre.


Comment entrer du texte pendant un débogage d'une application console ?
La fenêtre GDB vous permet de taper vos propres commandes.


Comment activer l'achèvement du code pour les bibliothèques externes ?
quand vous avez un projet C/C++ ouvert, allez dans le menu Projet->Paramètres du projet, dans la boîte de dialogue Options du projet choisissez l'onglet C++ Specific, ensuite l'onglet Code Completion. Presser le bouton Add Persistant Class Store et complétez l'assistant New Persistant Class Store. Après avoir complété l'assistant, assurez-vous que l'enregistement de la classe créée et coché dans la liste des classes persistantes enregistrées sur l'onglet Code Completion.

Je développe une application pour laquelle je dois activer les options -D au préprocesseur. Si je l'ajoute dans mes options de projet dans KDevelop, tout va bien, mais quand je distribue mon paquetage, les options -d activée dans KDevelop ne sont pas utilisées. Comment faire pour que ça marche ?
Éditez le fichier configure.in(.in) dans votre répertoire racine de votre code source. Là, vous
devez entrer après la macro AC_CHECK_COMPILERS :
CPPFLAGS="$CPPFLAGS -DMYDEFINE"
et après ça vous devez recréer votre nouveau "configure" en invoquant
"make -f Makefile.dist" et ensuite relancer le script configure.


Y a-t-il une autre possibilité pour publier mes propres defines ?
Une autre façon d'inclure des DEFINE dans votre projet est de les publier
avec config.h.
Pour ce faire vous devez patcher les fichiers suivants :
Ajouter la ligne suivante au fichier acconfig.h du répertoire racine de votre projet :
#undef MYDEFINE

et insérez dans le fichier configure.in(.in), apré l'appel de macro
AC_CONFIG_HEADER(config.h) :
AC_DEFINE_UNQUOTED(MYDEFINE)

(Ainsi, il sera défini dans config.h après la mise à jour de votre cadre de travail avec
« make -f Makefile.dist; configure [vos options ».

Les conditions nécessaires pour cette solution sont :
AC_CONFIG_HEADER(config.h)
dans configure.in(.in) et dans le code source un
#ifdef HAVE_CONFIG_H
#include <config.h>
#endif
avant que les autres includes ne se fassent
(normalement c'est déjà fait dans vous modèles de projet).

NOTE : Un avantage de cette solution est de rendre MYDEFINE dépendant de
certaines situations (comme créer un option pour configure).
Ca nécessite bien sûr l'écriture d'une fonction macro propre.


Je développe une application qui a besoin de gérer les exceptions. Comme faire ?
Éditez le fichier configure.in(.in) dans le répertoire racine de votre projet. Là, vous
devez entrer après la macro AC_CHECK_COMPILERS() :
CXXFLAGS="$CXXFLAGS $USE_EXCEPTIONS"
et après ça, vous devez recréer votre nouveau « configure » en invoquant
« make -f Makefile.dist » dans le répertoire racine du code source et et ensuite redémarrer
le script configure.

Les solutions ci-dessus sont bien belles, mais je veux cela pour un certain sous-répertoire (p.ex. une bibliothèque statique de mon projet) Comment faire ?
Éditez le fichier Makefile.am dans le répertoire source désiré. Là, vous
devez entrer quelque part en-dehors de la partie spécifique à KDevelop :
KDE_CXXFLAGS=$(USE_EXCEPTIONS)
Faites attention, ça ne marche que si votre projet utilise am_edit.
(Ca ne marche que pour les nouveaux acinclude.m4 & am_edit, AFAWK ils doivent être
plus récents que 09-12-1999, autrefois, c'était APPEND_CXXFLAGS -
dans cette version de KDevelop vous trouverez le bon.)

Vous devez mettre à jour votre environnement Makefile comme décrit plus haut
(make -f Makefile.dist; ./configure [options]).


Quelle est la différence entre CPPFLAGS et CXXFLAGS ?
CPPFLAGS est utilisé pour des defines (c'est-à-dire les drapeaux dont vous n'avez besoin que pour la compilation)
CXXFLAGS est utilisé pour des switches du compilateur (e.a. -frtti), et aussi
pour le processus de linkage.


J'ai vu plusieurs solutions comme patcher Makefile.am avec DEFS+=-DMYDEFINE ou CXXFLAGS+=-fsigned-char ou LD_FLAGS+=-L/usr/local/lib Pourquoi ne pas les considérer ?
Il y a un bogue dans automake 1.4, qui n'ajoute pas de drapeau à ceux
qui existent déjà. Avec cette commande dans Makefile.am, vous effaceriez les paramètres existants
du drapeau en question.
Si vous voulez ces drapeaux uniquement pour le sous-répertoire du projet, utilisez :
AM_CPPFLAGS=-DMYDEFINE
ou
AM_CXXFLAGS=-fsigned-char
ou
AM_LDFLAGS=-L/usr/local/lib

Attention, ces drapeaux sont seulement publiés par automake version 1.4.

Pour paramétrer un de ces drapeaux pour l'ensemble du projet, il faut patcher
le fichier configure.in(.in) dans le répertoire racine du code source à la place.


Comment être certain qu'un utilisateur de mon projet le construira avec automake version 1.4 ?
Ajoutez la variable AUTOMAKE_OPTIONS dans le fichier Makefile.am du répertoire racine
du code source avec la chaîne « 1.4 ».
P.Ex .: AUTOMAKE_OPTIONS = foreign 1.4


J'ai ajouté un sous-projet existant à mon projet et ça ne veut plus se construire.
Ça peut arriver si builddir != topsrcdir. Dans ce cas, exécutez make distclean avant
d'ajouter le sous-projet.


AM_CXXFLAGS (dans un Makefile.am) peut-il être utilisé comme drapeau comme -frtti ou -fexceptions ?
Non, dans ces cas-là il y a un problème d'ordre. La variable standard CXXFLAGS
publiée dans acinclude.m4.in contient déjà un -fno-rtti (-fno_exceptions).
Si vous l'avez appliquée à AM_CXXFLAGS la commande du compilateur insè alors
ces drapeaux avant les CXXFLAGS, p.ex. ici pour -frtti :
gpp ... -frtti ... -fno-exceptions -fno-rtti ......
Donc, le dernier drapeau (dans ce cas-ci, toujours -fno-rtti) sera utilisé.
KDE_CXXFLAGS par contre sera inséré après le CXXFLAGS standard, donc ça
devrait ressembler à ça :
gpp ..... -fno-exceptions -fno-rtti ...... -frtti...
et voilà ça marche.

Attention : KDE_CXXFLAGS sont seulement publiés par des projets qui utilisent am_edit
et acinclude.m4.


Pourquoi patcher configure.in(.in) avec CXXFLAGS="$USE_EXCEPTIONS $CXXFLAGS" ne fonctionne pas ?
Regardez le réponse à la question « AM_CXXFLAGS (dans un Makefile.am) peut-il être utilisé comme drapeau comme -frtti ou -fexceptions ? » ;-)


J'utilise CXXFLAGS="$USE_EXCEPTIONS" dans configure.in(.in) et ça marche bien, pourquoi utiliserais-je votre solution ?
Si vous utilisiez votre version avant AC_CHECK_COMPILERS(), un appel comme
CXXFLAGS="-pedantic" ./configure
ne fonctionnerait pas correctement.
Si vous l'utilisiez après AC_CHECK_COMPILERS(), le CXXFLAGS affecté dans
AC_CHECK_COMPILERS() serait effacé.
(Par exemple dans un appel « configure --enable-debug », le CXXFLAGS serait
changé pour insérer aussi les informations de debogage à votre code.)
Donc, veuillez utiliser CXXFLAGS="$CXXFLAGS $USE_EXCEPTIONS" à la place.


J'utilise des entêtes et des bibliothèques supplémentaires pour mon projet KDE/QT. Mettre
-I/usr/include/foo dans les options du compilateur et -L/usr/lib/foo dans les options du lieur
ne transmet pas les paramètres dans le tarball.
Comment faire ?
C'est la partie la plus difficile à faire convenablement.
Une solution facile mais incorrecte serait d'ajouter dans le fichier configure.in(.in) :

all_includes="$all_includes -I/usr/include/foo"
all_libraries="$all_libraries -L/usr/lib/foo"

[le meilleur endroit pour ça, c'est avant les lignes :
AC_ARG_PROGRAM
AC_OUTPUT( ..... )
à la fin du fichier.]

Les problèmes ici :
- Qui peut garantir que les includes seront toujours dans
/usr/include/foo... peut-être que sur certains systèmes, ça pourrait être dans /usr/local/foo/include
(et peut-être que les bibliothèques seraient dans /usr/local/foo/lib) ?
- Peut-être que je n'ai pas installé les choses foo, alors la construction échouerait, mais ce serait
mieux si configure avait déjà échoué avec un message plus utile.

Donc, pour le faire convenablement, vous devez ajouter une macro à
configure.in(.in), qui fait certains tests et cherche un « foo » installé.

Si vous voulez le faire comme ça, nous vous donnons un truc... essayez de
comprendre le fichier acinclude.m4.in ou créez un projet KDE-normal-ogl,
où vous trouverez un fichier qgl.m4, qui sera concaténé à
acinclude.m4 (voyez "Makefile.dist"). Ca vous montre un exemple étendu
pour chercher QT-OpenGL-Lib, vous devez le réécrire pour votre problème.

Peut-être que vous avez de la chance et que vous avez déjà une solution à votre problème dans
acinclude.m4.in ainsi vous pouvez utiliser une fonction déjà créée.

Comment créer un simple projet avec seulement un Makefile (aussi connu sous le nom Projet Personnalisé) ?
Créez un projet personnalisé. Il n'y a pas de manière directe de créer un projet personnalisé (p.ex. un projet qui fait ses propres makefiles). Utilisez Project->Import Existing Project à la place. N'oubliez pas d'affecter le type de projet approprié, marqué par un  additionnel (Custom Makefiles), dans la boîte de dialogue. Utilisez les options du projet tôt. A chaque fois que vous commencez un nouveau projet, n'oubliez pas de régler le Project->Project Options... à vos besoins.

Pourquoi je ne vois pas les messages d'erreur de compilation ?
Si pendant les compilations, vous remarquez qu'une information importante est manquante dans la fenêtre de vue des messages de sortie, ça peut être que le niveau de détail des messages est réglé trop bas. Faites un clic-droit dans la fenêtre et sélectionnez un aute niveau de détail dans le menu contextuel.

Comment créer des nouveaux fichiers dans un projet ?
Sélectionnez le sous-projet dans la moitié supérieure de Automake Manager, ensuite clic-droit dans la partie inférieure sur le titre des groupes pour lesquels vous voulez ajouter les fichier et sélectionnez Créer un nouveau fichier....

Comment ajouter des fichiers existants à un projet ?
Sélectionnez le sous-projet dans la moitié supérieure de Automake Manager, ensuite clic-droit dans la partie inférieure sur le titre des groupes pour lesquels vous voulez ajouter les fichier et sélectionnez Ajouter des fichiers existants....

Comment supprimer un fichier d'un projet ?
Sélectionnez le sous-projet dans la moitié supérieure de Automake Manager, ensuite clic-droit dans la partie inférieure sur le titre des groupes pour lesquels vous voulez supprimer les fichier et sélectionnez Supprimer....

Que faire si un projet ne veut plus se construire après avoir changé de/vers la cible par défaut ?
Il y a un problème avec la machinerie automake/autoconf. Si Project->Build Configuration vous permet de choisir parmi trois répertoires de construction : par defaut, optimisé, et debug, ça veut dire que vous choisissez l'un ou l'autre des cibles « par défaut » ou debug/optimisé.
Une fois que vous avez configuré votre projet avec par « défaut », il ne se construira plus avec debug ou optimisé.
Une fois que vous avez configuré votre projet avec debug ou optimisé, il ne se construira plus avec « par défaut ».


Erreur « Mauvaise version autoconf» etc.
Il y a quelques messages d'erreur concernant des trop vieilles versions d'autoconf etc. empêchant configure de marcher convenablement. Exécutez autoreconf dans le répertoire où se trouvent les fichiers configure.in en question. Cette commande essayera de mettre à jour l'information dans les fichiers du système de construction GNU. Voyez man autoreconf pour plus d'informations.
