Où puis-je trouver KDevelop ? Comment l'installer ?
L'équipe KDevelop ne fournit pas versions binaires de KDevelop (RPMs). Les RPMs peuvent être trouvés sur le serveur de votre distribution ou dans les sous-répertoires correspondants de ftp.kde.org ou l'un de ses miroirs
Veuillez regarder notre page de téléchargements pour plus d'information sur comment télécharger et compiler le code source.


Puis-je développer des applications commerciales avec KDevelop ?
Chacun et toute entité est libre d'utiliser KDevelop. Le code généré par KDevelop peut être mis sous toute license que vous voulez. Ainsi, le générateur de projet possède des champs pour ajouter votre propre copyright aux fichiers générés, mais, pour rendre les choses plus simples et pour donner un exemple, l'option par défaut est une notification de license GPL. Vous pouvez donc ajouter votre notification de license là.
Pour ce qui est d'éditer etc. votre programme, utiliser KDevelop est simplement comme utiliser tout autre éditeur, c'est un programme qui est GPL pour ses utilisateurs et cela n'a aucune implication sur pourquoi vous voulez l'utiliser. Vous pouvez ainsi écrire des logiciles commerciaux/fermés avec, en dépit du fait que KDevelop lui-même est GPL.
Pour ce qui est d'utiliser les bibliothèques KDE ou toute autre partie d'un logiciel libre, il vous faut seulement regarder les informations sur la license des bibliothèques que vous allez utiliser. Ainsi, pour une utilisation commerciale d'applications KDE/Qt, vous n'avez besoin que d'une license professionnelle de Trolltech. Les bibliothèques KDE en elles-mêmes sont LGPL ce qui signifie que vous pouvez utiliser les appels de fonctions et lier vos programmes aux bibliothèques KDE. Ceci est valable pour les bibliothèques kdelibs, les bibliothèques kdebase (konqueror et kcontrol etc.) et les bibliothèques koffice (pour créer des applications koffice).
Pour plus d'information, veuillez voir license de KDevelop


Est-il permis de développer des modules externex commerciaux pour KDevelop ?
KDevelop inclut une bibliothèque libkdeveloplgpl.so pour permettre le développement de modules externes commerciaux. Cette bibliothèque compremnd les interfaces KDevelop et les fonctions utilitaires spubliées sous LGPL. Si vous avez une licence professionnelle de Trolltech vous pouvez lier votre code avec libkdeveloplgpl et le publier sous n'importe quelle licence.


J'ai compilé et installé gideon mais je ne vois seulement qu'un menu Fichier et un menu Aide
Comme pour toute autre application KDE, vous devez être sur que les plugins de KDevelop sont reconnus par KDE. Installez-le soit dans votre répertoire KDE (configure --prefix=/opt/kde3 par exemple) ou ajouter le chemin vers gideon à votre variable d'environnement KDEDIRS et relancez "kbuildsycoca". Un exemple de commande pour bash (en supposant que gideon est installé dans /home/harry/gideon) :
export KDEDIRS=/home/harry/gideon:$KDEDIRS && kbuildsycoca


Comment puis-je naviguer depuis le clavier dans des fichiers ouverts ?
Vous pouvez utiliser soit de CTRL-1 jusqu'à CTRL-0 pour naviguer entre les dix premiers fichiers soit ALT-PageUp et ALT-PageDown pour aller au prochain/précédent fichier ouvert. Ces touches peuvent être configurées dans l'écran de configuration de KDevelop.


 Comment puis-je entrer du texte pendant que je débogue une application en console ?
La fenêtre de sortie StdOut Window est en lecture seule, vous devez ouvrir une autre console en mettant une croix dans Options->Configure KDevelop ->Débogueur->Permet un terminal séparé pour entrées-sorties.


 Comment puis-je configurer la complétion de code pour des bibliothès externes ?
Lorque vous avez ouvert un projet C/C++, allez dans le menu Projet->Configurations du projet, dans le dialogue des options du projet choisissez l'onglet spécifique C++, puis Complétion de codepuis le boutonAjouter Persistant Class Store et complétez l'assistant New Persistant Class Store. Lorsque l'assistant est terminé, veuillez vous assurer que la nouvelle class store est cochée dans la liste des persistant class stores dans l'onglet Complétion de code.

 Éditez le fichier configure.in(.in) du répertoire source supérieur de
votre projet. Vous devez
ajouter ici après la macro AC_CHECK_COMPILERS :
CPPFLAGS="$CPPFLAGS -DMYDEFINE"
Ensuite, vous devez recréer votre nouveau "configure" en lançant
"make -f Makefile.dist" puis relancer le script de configuration
(ou, sous KDevelop utilisez "Build/Autoconf and Automake" et
"Build/Configure...").


 Y a-t-il une autre possibilité pour publier mes propres "define" ?
 Une possibilité pour inclure DEFINEs dans votre projet est de les publier
avec config.h.
Pour cela, vous devez patcher les fichiers suivants :
ajouter au fichier acconfig.h du répertoire principal de votre projet la
ligne suivante :
#undef MYDEFINE

et insérer dans le fichier configure.in(.in), après l'appel de la macro
AC_CONFIG_HEADER(config.h):
AC_DEFINE_UNQUOTED(MYDEFINE)

(Il sera ainsi défini dans in config.h après la mise à jour de votre cadre de
travail avec
"make -f Makefile.dist; configure [vos options]"
ou bien dans KDevelop en utilisant "Build/Autoconf and Automake" et
"Build/Configure...").

Sont nécessaires pour cette solution :
AC_CONFIG_HEADER(config.h)
dans configure.in(.in) et dans le code source un
#ifdef HAVE_CONFIG_H
#include
#endif
avant que les autres en-têtes soient mentionnés
(ceci est normalement déjà inscrit dans les squelettes du projet).

NOTE : Un avantage de cette solution est de rendre MYDEFINE dépendant de
certaines situations (comme créer une option pour configure).
Cela nécessite bien sûr d'écrire sa propre fonction macro.

 Quelle est la différence entre CPPFLAGS et CXXFLAGS ?
 CPPFLAGS est utilisé pour les defines (pour des drapeaux nécessaires
seulement pour la compilation)
CXXFLAGS est utilisé pour des interrupteurs de compilateur (par exemple
-frtti), ce qui s'applique aussi
au processus de lien.


 J'ai vu certaines solutions comme de patcher le fichier Makefile.am avec
DEFS+=-DMYDEFINE
ou
CXXFLAGS+=-fsigned-char
ou
LD_FLAGS+=-L/usr/local/lib
POurquoi ne proposez-vous pas cela ?
 Il y a un bogue dans automake 1.4, qui n'ajoute pas de drapeaux à ceux qui
existent déjà.
Si vous utilisez cette commande dans Makefile.am, vous supprimeriez la
configuration du drapeau
existant.
Si vous voulez ces drapeaux uniquement pour le sous-répertoire du projet,
utilisez :
AM_CPPFLAGS=-DMYDEFINE
ou
AM_CXXFLAGS=-fsigned-char
ou
AM_LDFLAGS=-L/usr/local/lib

Faites attention, ces drapeaux sont uniquement disponibles avec automake
version 1.4.

Pour configurer l'un de ces drapeaux pour le projet entier, veuillez patcher
le fichier
configure.in(.in) dans le répertoire principal de votre projet.

 Est-ce que AM_CXXFLAGS (dans un Makefile.am) peut être utilisé comme drapeau
comme -frtti ou
-fexceptions ?
 Non ! Dans ces cas, il y a un problème d'ordre. La variable standard
CXXFLAGS publiée
par acinclude.m4.in contient déjà un -fno-rtti (-fno_exceptions).
Si vous l'appliquez à AM_CXXFLAGS la commande du compilateur voudrait insérer
ces
drapeaux avant CXXFLAGS, c'est à dire ici pour -frtti:
gpp ... -frtti ... -fno-exceptions -fno-rtti ......
Ainsi le dernier drapeau (dans ce cas, toujours -fno-rtti) serait utilisé.
KDE_CXXFLAGS au contraire sera inséré après le CXXFLAGS standard, et il
ressemblera à :
gpp ..... -fno-exceptions -fno-rtti ...... -frtti...
et cela marche bien.

Faites attention : KDE_CXXFLAGS ne sont publiés que pour les projets qui
utilisent am_edit
et acinclude.m4.


 Pourquoi le patch de configure.in(.in) avec CXXFLAGS="$USE_EXCEPTIONS
$CXXFLAGS" ne fonctionne-t-il pas ?
 Regardez la réponse à la question "Est-ce que AM_CXXFLAGS (dans Makefile.am)

 J'utilise CXXFLAGS="$USE_EXCEPTIONS" dans configure.in(.in) et cela
fonctionne bien,
pourquoi devrais-je utiliser votre solution ?
 Si vous utilisiez votre version avant AC_CHECK_COMPILERS() un appel comme
CXXFLAGS="-pedantic" ./configure
ne fonctionnerait pas bien.
Si vous l'utilisiez après AC_CHECK_COMPILERS() le CXXFLAGS configuré dans
AC_CHECK_COMPILERS() serait annulé.
(Par exemple dans un appel à "configure --enable-debug", CXXFLAGS serait
changé pour aussi insérer des informations de déboguage dans votre code.)
Veuillez donc utiliser CXXFLAGS="$CXXFLAGS $USE_EXCEPTIONS" à la place.


 J'utilise des en-têtes et des bibliothèques supplémentaires dans mon
projet KDE/Qt. Si je mets
-I/usr/include/foo dans les options de compilation et -L/usr/lib/foo dans les
options de lien, cela ne va pas
fixer cetrte configuration dans le paquetage.
Que dois-je faire ?
 C'est la chose la plus difficile à faire correctement.
Une solution facile mais incorrecte serait d'ajouter dans le fichier
configure.in(.in):

all_includes="$all_includes -I/usr/include/foo"
all_libraries="$all_libraries -L/usr/lib/foo"

[le meilleur endroit pour cela est avant les lignes :
AC_ARG_PROGRAM
AC_OUTPUT( ..... )
à la fin du fichier.]

Les problèmes sont :
- Qui peut garantir que les en-têtes seront topujours dans
/usr/include/foo... Peut-être que dans certains systèmes ils seront dans
/usr/local/foo/include
(et peut-être que les bibliothèques seront dans /usr/local/foo/lib) ?
- Peut-être que foo n'est pas installé et la compilation va échouer mais ce
serait mieux si
configure échouait déjà avec un message d'aide plus indicatif.

Aussi prenez la peine de faire cela correctement si vous devez ajouter une
fonction macro à
configure.in(.in), qui effectue certains tests et cherche si "foo" est
installé et où.

Si vous voulez faire ainsi, nous ne pouvons que vous donner quelques pistes
ici... essayez de comprendre
le fichier acinclude.m4.in ou créez un projet KDE-ogl normal.
Vous trouverez un fichier appelé qgl.m4, qui sera transformé en
acinclude.m4 (voir "Makefile.dist"). Ceci vous montre un exemple de recherche
de
QT-OpenGL-Lib, vous devrez donc le récrire pour votre cas particulier.

Vous aurez peut-être de la chance et trouverez une solution à votre problème
dans
acinclude.m4.in, et vous pourrez utiliser une fonction déjà créée.
