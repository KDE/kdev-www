Bestands FAQ

Waar is de AutoMake manager in KDevelop?

Klik op de onderste tab aan de rechterkant van KDevelop.


Hoe creëer ik bestanden in een project?

Selecteer het sub-project in de bovenste helft van de AutoMake manager, rechts-klik dan in de onderste helft op de groepstitel waar je het bestand wil toevoegen en selecteer Create New File....


Hoe verwijder ik een bestand uit een project?

Selecteer het sub-project in de bovenste helft van de AutoMake manager, open dan in de onderste helft de groeplijst waar je een bestand van wil verwijderen, rechts-klik dan op de file in deze lijst en selecteer Remove.


Hoe voeg ik bestaande bestanden aan een project toe?

De manier waarop je bestaande bestanden toevoegd aan een project is afhankelijk van het bouwsysteem dat je gekozen hebt voor je project. Echter zal je, ongeacht welk bouwsysteem je ook gekozen hebt, de bestaande code-bestanden in de src/ map van je nieuwe KDevelop project moeten zetten.

Next:

  • Als je hebt gekozen voor een AutoMake manager gebaseed project (zie: FAQ#Waar is de AutoMake manager in KDevelop?) dan voeg je bestaande bestanden toe aan je project door het goede sub-project, in de bovenste helft van de AutoMake manager, te selecteren. Dan in de onderste helft de groep aanklikken van het doel waar je de bestanden aan wil toevoegen. Rechts-klik erop voor het context menu en selecteer "Add Existing files...."
  • Als je hebt gekozen voor een QMake manager gebaseerd project dan is je taak hetzelfde als bij de AutoMake manager.
  • Als je hebt gekozen voor een op een eigengemaakte Makefile gebaseerd project dan moet je je Makefile zelf bewerken om de bestanden toe te voegen aan je project.


Project FAQ

Hoe converteer ik mijn VC++ project?

KDevelop heeft geen manier om je VC++ project automatisch te importeren. Je kan echter wel je project converteren door een nieuw KDevelop project via de voorbeelden van de "New Project" wizard te maken, zoals het C++->Simple Hello World voorbeeld. Dan kan je de code bestanden (*.cpp en *.h) kopieëren van je VC++ project naar de src map van je nieuwe KDevelop AutoMake gebaseerde project. Volg als laatste de instructies hierboven voor het toevoegen van bestaande bestanden aan een project.

Een mindere oplossing kan door het gebruik van de dsw2make script, Deze pakt een VC++ project bestand (*.dsw) met zijn onderliggende dsp bestanden en proceduceerd een Makefile voor iedere dsp. Controleer of het werkt want fouten kunnen regelmatig optreden. Deze hebben vaak te maken met de paden naar de bestanden, hoofdlettergevoeligheid en CR/LF. Als een CR/LF probleem ontstaat gebruik applicaties als dos2unix of d2u om deze problemen op te lossen. Hierna zou je een normaal project moeten hebben gebaseerd op een makefile-ketting. Import het project als een bestaand project.

Hoe creëer ik een totaal leeg project?

  • Ga naar Project->New Project...->C++->Simple Hello World program en creëer een project
  • Verwijder het enige bestaande doel in het project
  • Selecteer remove also source files in het dialoog dat tevoorschijn komt.


Hoe creëer je een simpel project met alleen een makefile (ook bekend als een custom project) ?

Creëer een custom project. Er is geen direkte manier om een custom project te maken (d.w.z. een project dat zijn eigen makefiles gebruikt). Gebruik Project->Import Existing Project daarvoor. Vergeet niet om het goede projecttype te zetten, gelabeled door een extra (Custom Makefiles) in het dialoog. Gebruik Project Options vroeg in het project. Wanneer je start met een nieuw project vergeet niet om de Project->Project Options... te zetten zoals je hem dan weer nodig hebt.


Hoe creëer je een simpel AutoMake project met een mappen hiërarchie ?

Als je een project wil creëeren met een mappen hiërarchie, waar iedere map een groep van cpp/h bestanden bevat die moeten worden gecompileerd naar een enkele binair bestand. 

Ex:
  src
  src/gui
  src/gui/components
  src/logic
  src/logic/algorithms
  src/logic/sublogic


Dit kan voor elkaar gekregen worden door het gebruik van bibliotheken. Iedere subdirectory zal worden gecompileerd en gelinked tot een libtool-bibliotheek, hierna worden alle libtool-bibliotheken gelinked tot een executable.

  • Stap 1:
    • Voeg in de AutoMake manager een subproject toe voor ieder submap:
    • AutoMake manager -> [rechts-klik op submap] -> Voeg Subproject toe
  • Stap 2:
    • Voeg een Libtool doel voor ieder project toe, en vul deze bibliotheken met de .cpp bestanden in de submap:
      • AutoMake manager -> [rechts-klik op submap] -> Voeg doel toe
      • Primary: Libtool Library
      • Prefix: noinst
      • File name: <je naam>.la
  • Stap 3:
    • Voeg de bibliotheken to aan je executable:
      • AutoMake manager -> [rechts-klik op binair] -> Options -> Libraries
  • Stap 4:
    • Voeg de afhankelijkheden van de bibliotheken toe voor ieder subproject:
    • Als libalgo.la afhankelijk is van klasses in libbase.la:
      • AutoMake manager -> [rechts-klik op libalgo] -> Options -> Libraries (vink de box voor libbase.la aan) 
Ex:
 src/logic/base [libbase.la]
 src/logic/algorithm [libalgo.la]

Let op: Als je een afhankelijkheid vergeet, voeg de afhankelijkheid toe, clean je subproject, en hercompileer het gehele project (F8).


Wat betekend de "build configurations" (Project Options->Configure Options)?

  • debug:
    • De aparte bouwmap is "debug"
    • -g3 -O0 compiler opties worden gebruikt om debuggen mogelijk te maken zonder problemen met optimalizatie
  • optimized:
    • De aparte bouwmap is "optimized"
    • -O2 -g0 flags om debuggen af te zetten en optimalizatie aan te zetten
  • default:
    • Geen speciale bedoeling, geen compilervlaggen en bouwmap
    • gecompileerde programma's en bibliotheken worden bij de code bestanden neergezet
    • Belangrijk! Gebruik niet de "default" configuratie samen met "debug" en "optimized". Maar "debug" en "optimized" kunnen samen wel gebruikt worden zonder problemen omdat er verschillende bouwmappen gebruikt worden.


Wat moet ik doen als het programma niet meer compileerd na het wisselen van/naar de standaard compileertype?

Er is een probleem met de automake/autoconf programma's. De Project->Build Configuration geeft de mogelijkheid om uit drie verschillende bouwmappen te kiezen: default, optimized, en debug, zorg dat je of default of debug/optimized doelen kiest.

  • Wanneer je je project op default hebt geconfigureerd zal hij niet meer met debug of optimized kunnen bouwen.
  • Wanneer je je project op debug of optimized hebt geconfigureerd zal hij niet meer met default kunnen bouwen.


Hoe kan ik subprojecten toevoegen?

Selecteer de AutoMake manager. In de bovenste helft zie je de projecten. Om een subproject toe te voegen selecteer het project aan welke je een subproject wil toevoegen en klik op het icoon "add subproject". Een nieuwe map is gecreëerd met de naam van het subproject (in het bestandssysteem en in de AutoMake manager).

Je moet nu een libtool-doel creëeren voor het nieuwe subproject. Als je dit hebt gedaan maak deze actief en voeg er bestanden aan toe.

Het libtool-doel in het subproject kan worden gelinked aan je hoofd binair doel door rechts-klikken op dit doel en selecteer "settings". Onder de tab "libraries" kan je het goede hulpbibliotheek (.la bestand) selecteren.

Wat moet ik doen als automake & friends breken op rare foutmeldingen?

Open een consolescherm in je project root-map en typ: 

libtoolize --copy --force

Dit zou het probleem van conflicterende autotool-versies tussen KDevelop en die op je systeem geinstalleerde moeten oplossen.


Ik krijg "Wrong autoconf version" etc. Error. Wat kan ik doen?

Er zijn verschillende foutmeldingen over te oude versies van autoconf e.d. die ervoor zorgen dat configure niet goed werkt. Start autoreconf in de mappenstructuur waar de configure.in bestanden in staan, waarop de foutmeldingen worden gegeven. Dit commando zal probleren de informatie in de "GNU Build System"-bestanden te updaten. Zie man autoreconf voor meer informatie.


Hoe kan ik ontwikkelen met Qt4 en Kdevelop-3.4.x?

  • Ga naar de Project Opties (in het Project menu) en selecteer de C++ Support. Open de Qt tab en

controleer of de Qt4 checkbox aangevinkt is. Kies dan de basisdirectory van jou Qt4 installatie, bijvoorbeeld: 

/usr/share/qt4 (op een Debian Unstable Systeem)
  • Als je qmake gebruikt, kan je de benodigde Qt bibliotheken voor je project kiezen via de qmake manager. Open deze en rechts-klik op een subdirectory of de basisdirectory en kies opties van het contextmenu. Op de Configuratiepagina zie je de verschillende Qt4 bibliotheken en je kan degene kiezen die je nodig hebt


Hoe kan ik ontwikkelen met Qt4 en Kdevelop-3.3.x?

  • Wanneer je qmake gebruikt, gebruikt KDevelop qmake van het PATH, dus de qmake van Qt4 moet eerst in het PATH gezet worden (http://bugs.kde.org/show_bug.cgi?id=119107). Om dit te doen moet je KDevelop niet zoals normaal opstarten, maar het volgende shellscript gebruiken: 
#!/bin/sh
export QTDIR=/devel/kde/src/4/qt-copy
export PATH=$QTDIR/bin:$PATH
exec kdevelop $*
  • Het tweede probleem is dat de ingebouwde Qt designer is die van Qt3. Inplaats hiervan willen we de Qt4 designer starten voor de .ui-bestanden. Hier zijn twee stappen voor nodig:

Open ~/.kde/share/config/kdeveloprc en zoek de [General Options] groep (niet [General]!), en voeg de volgende lijn toe: 

Embed KDevDesigner=false

Creëer ~/.kde/share/applnk/.hidden/designer.desktop met de volgende inhoud (verander het pad naar Qt4) 

[Desktop Entry]
Type=Application
Exec=/devel/kde/src/4/qt-copy/bin/designer
Name=Qt Designer
GenericName=Interface Designer
MimeType=application/x-designer
Icon=designer

Het volledige pad naar de designer is hier nodig, zelfs alhoewel het PATH is correct in KDevelop. Dit is zo omdat KDevelop de designer niet direct start, het vraagt KDEinit om dit te doen...

  • Gebruik het "Hello World"-project wanneer je een nieuw project maakt, niet "Application", anders zal je eerste Qt4 project een Qt3->Qt4 porteer project zijn :)


Hoe zet ik een project op met SCons?

SCons is een software constructie hulpmiddel. KDevelop (vanaf versie 3.2.2) ondersteund SCons-projecten direct. Het is echter mogelijk om een project op te zetten die gebruik maakt van SCons door een "stub" Makefile te maken en gebruik te maken van de "Custom Makefiles" project type. Een voorbeeld van een simpele stub Makefile is het volgende (Let op dat de regels achter de target-namen met tabs begonnen moeten worden, niet met spaties): 

project_name:
        scons
clean:
        scons --clean

Dit zorgt dat de Build Project en Clean Project acties werken.


Libraries FAQ

Hoe maak ik een .so-bibliotheek?

  • Voeg een doel toe aan het project
    • primary: program
    • prefix: bin
    • file name: libxxx
    • Linker flags->Other: -shared -Wl,-soname,libxxx.so.1 -o libxxx.so.1.0.1
  • Voeg de cpp/h bestanden naar dit doel
  • zet de compiler vlaggen voor c++ -fPIC

Er is ook een libtool gebaseerde methode:

  • Voeg een sub-project "libxxx" toe
  • Voeg een doel toe
  • kies "Library (libtool)"
  • kies "lib"
  • Zet als bibliotheeknaam "xxx"
  • Voeg je bestanden toe
  • Compileer het project (Gebruik 'Build project' of F8).

Nu zou je een nieuwe bibliotheek moeten hebben genaamd libxxx.so.0.0.0



Hoe maak ik een .a-bibliotheek?

  • Voeg een doel toe aan het project
    • primary: Library
    • prefix: noinst
    • file name: jouw_bibliotheek
  • Voeg de cpp/h bestanden aan dit doel


Hoe maak ik een .la-bibliotheek (ook bekend als een libtool-archief)?

  • Voeg een doel toe aan het project
    • primary: Libtool Library
    • prefix: noinst
    • file name: jouw_bibliotheek
  • Voeg de cpp/h bestanden aan dit doel


Hoe voeg ik een extern include-pad to aan een project?

Wanneer je project gebaseerd op GNU Autotools (Autoconf/AutoMake) dan gebruik je waarschijnlijk de KDevelop AutoMake manager (zie FAQ#Waar is de AutoMake manager in KDevelop?). Om een include-pad toe te voegen aan je project doe je:

  • selecteer het subproject waar je het pad aan toe wil voegen in de bovenste helft van de AutoMake manager.
  • rechts-klik op Options... of klik op het sleutel-icoon
  • in de Includes tab voeg je het pad toe in de tweede box (Directories outside of project). Voeg het pad toe na de -I (zonder een spatie ertussen).


Hoe link ik tegen een .so-bibliotheek aan?

Wanneer je project gebaseerd op GNU Autotools (Autoconf/AutoMake) dan gebruik je waarschijnlijk de KDevelop AutoMake manager (zie FAQ#Waar is de AutoMake manager in KDevelop?). Om te linken tegen een bibliotheek, of een bibliotheek toe te voegen doe je :

  • selecteer het subproject waar je de bibliotheek wil toevoegen in de bovenste helft van de AutoMake manager.
  • Rechts-klik op Options... of klik op het sleutel-icoon
  • in de libraries ("compiler") tab voeg je de bibliotheek toe in de tweede box (link libraries outside of project). Voer de bibliotheeknaam, zonder de prefix en de .a of .so of .la of waarop het ook eindigd
    • d.w.z. als je bibliotheek libsomelib.so heet, dan moet je toevoegen: -lsomelib
  • Als de bibliotheek geen onderdeel is van het bibliothekenpad van je systeem dan voeg je -Lpad/naar/de/bibliotheek/dat/je/gebruikt toe aan LDFLAGS in je Project->Project Options...->Configure Options->Linker Flags(LDFLAGS):.

Vergeet niet om de locatie van de bibliotheek zijn .h-bestand aan de includepad van je gecompileerde programma zoals is uitgelegd in FAQ#Hoe voeg ik een extern include-pad to aan een project?


Hoe link ik tegen een .a-bibliotheek aan?

Dit is hetzelfde als FAQ#Hoe link ik aan een .so-bibliotheek?


Hoe link ik tegen een .la-bibliotheek aan?

Dit is hetzelfde als FAQ#Hoe link ik aan een .so-bibliotheek?

Toevoeging: Er is een andere, meer voor de hand liggende, manier om mijnbinaire tegen eenbibliotheek.la te linken. Zet eenbibliotheek.la in $(bepaaldedir), dan, in plaats van -lsomestuff -L$(somedir) toe te voegen, voeg $(bepaaldedir)/eenbibliotheek.la toe. Het libtool programma kan hiermee omgaan.

Het volledige proces (zoals uitgelegd hierboven):

  • selecteer het subproject waar je de bibliotheek wil toevoegen in de bovenste helft van de AutoMake manager.
  • Rechts-klik op Options... of klik op het sleutel-icoon
  • in de libraries tab, verwijder de -l en voeg $(bepaaldedir)/eenbibliotheek.la toe in de tweede box (link libraries outside of project).

Dit is hetzelfde proces/behandeling als linken tegen bibliotheken binnen hetzelfde project, je moet de goede bibliotheken in de bovenste box (Link convenience libraries inside project) invoeren.

Enkele details: De bibliotheekfuncties zijn toegevoegd aan mijnbinaire_LDADD in de goede Makefile.am. De LDADD variabele kan libtool archieven afhandelen, d.w.z. het leest de locaties van de actuele bibliotheken en voegt ze toe.


Debug FAQ


Het debuggen van een console-applicatie : Hoe kan ik de console zien zodat ik data kan invoeren?

Ga in het Project Options dialoog naar de debugger tab en controleer Enable separate terminal for application IO

Hoe debug je exec-objecten met libtool "temporary wrapper scripts"?

Bij het bouwen van ltlibraries en het linken van de exec met hen, creëert libtool warppers om hen heen, maar gdb geeft not in executable format: File format not recognized

Om het werkende te maken doe:
Project -> Project Options -> Debugger -> Debugging shell -> libtool --mode=execute


Code Completion FAQ

Hoe activeer ik code completion?

Wanneer je een C/C++ project opent, ga naar menu Project->Project Options, in de Project Options dialoog kies C++ Specific tab, dan de Code Completion tab. Vink de Automatic Code Completion vinkbox aan en andere gerelateerde instellingen. Ook in de Settings->Configure KDevelop->C++ Parsing dialoog kan je dit aanzetten en de parsing loop aanpassen.


Hoe leer ik de C/C++ parser nieuwe macro's?

KDevelop's huidige parser parsed de bestanden niet in een bepaalde volgorde. Als je nieuwe marco's aan de C/C++ parser wil leren kan je dit het beste doen door een bestand dat alleen marco-definities bevat parsen voor de code-bestanden. Dit is waar de C++ Parser Settings dialoog voor is. De inhoud van dit bestand is de tekst dat je bewerkt in de C++ Parser Settings dialoog, die als eerste wordt geparsed wanneer je een project laad.


Hoe activeer ik code completion voor externe bibliotheken?

Wanneer je een C/C++ project hebt geopend, ga je naar het menu Project->Project Settings, in de Project Options dialoog en kiest de C++ Specific tab, dan de Code Completion tab. Druk op de Add Persistent Class Store knop en maak de New Persistent Class Store wizard af. Als je hiermee klaar bent zorg dan dat de nieuwe class store is aangevinked in de lijst van Persistent class stores in de Code Completion tab.


Code completion werkt nog steeds niet

Probeer het deactiveren van het afkortingen gedeelte - deze presenteerd soms een completion pupop voor de completion engine een kans krijgt om dat te doen.


Waarom werkt mijn Qt4 Persistent Class Store niet voor alle Qt4 klassen?

Wanneer je KDevelop 3.3.x of lager gebruikt, de persistent class store zal niet correct werken wanneer je Qt4's include directories importeerd. Qt4's includes zijn over het algemeen one-liners met '#include "../../src/blah.h", welke KDevelop niet kan behandelen. Om hier omheen te werken, probeer de QT4's src directory te gebruiken wanneer je de completion database maakt.


KDevelop installatie FAQ

Hoe lang duurt het compileren van KDevelop normalieter?

Dit hangt af van de snelheid van je computer. Een ontwikkelaar met een 3GHZ P4 doet er omgeveer 40 minuten om KDevelop te compileren voor de eerste keer. Na het updaten van CVS duurt het meestal minder lang. Wij gokken dat het gemiddeld tussen 1 en 2 uur duurt om te compileren na de eerste download.

Het is een goed idee om gcc-4.01 te gebruiken. Dit zal de compileertijd drastisch verminderen. Debug compilaties ("./configure --enable-debug=full" met -g -O0 vlaggen) zijn 45% sneller en release ("./configure --enable-debug=no" met -O2 vlag) - zijn 16% sneller. Bijvoorbeeld, op een AthlonXP1600 / 256Mb RAM systeem een nieuwe debug compilatie duurt 45 minuten met gcc-4 en 81 minuten met gcc-3-3-1. Om het allemaal nog sneller te maken kan ook de meest recente libtool (van CVS HEAD) worden gebruikt.

Het is normaal veilig om gcc-4 geïnstalleerd te hebben in /usr/local samen met gcc-3.x in /usr. Het is ook mogelijk om alleen KDevelop met gcc-4 te compileren en de rest (KDE/Qt4) niet.

Let op! Gebruik geen gcc-4.0.0 want deze compileerd de code verkeerd. gcc-4.0.1 en latere versies hebben dit probleem niet.


Ik krijg de foutmelding "couldn't create a language plugin". Wat is er fout?

Net zoals ieder andere KDE applicatie moet je er zeker van zijn dat KDevelop's plugins herkent worden door KDE. Of installeer het in je KDE directory (configure --prefix=/opt/kde3 for example) of voeg het KDevelop pad toe aan je KDEDIRS omgevingvariabele en herstart "kbuildsycoca". Een voorbeeld voor het commando in bash (ervanuitgaand dat Kdevelop is geïnstalleerd in /home/harry/kdevelop): 

export KDEDIRS=/home/harry/kdevelop:$KDEDIRS && kbuildsycoca


Licentie FAQ

Ben ik bevoegd om commerciële applicaties te ontwikkelen met KDevelop?

Iedereen en alle entiteiten zijn vrij om KDevelop te gebruiken. De gegenereerde Code kan onder alle licenties die je wil worden ondergebracht. Daarom heeft de projectgenerator velden om je eigen copyright notice aan de gegenereerde bestanden toe te voegen, maar om het simpeler te maken en om een voorbeeld te hebben is er standaard de GPL licentie. Dus je kan daar je eigen licentie neerzetten. Als het gaat om het eigenlijke bewerken etc. van je programma is KDevelop net zoals een andere editor, het is een programma dat GPL is en geen implicaties heeft op waar je het voor gebruikt. Dus kan je er commerciële/gesloten software mee schrijven ondanks het feit dat KDevelop zelf GPL is. Voor het gebruiken van KDE functies of andere vrije software dingen, moet je alleen letten op de licentievoorwaarden op de bibliotheken die je gaat gebruiken. Dus voor commerciële ontwikkeling van KDE/Qt applicaties heb je alleen een professionele licentie nodig van Trolltech. De KDE bibliotheken zijn zelf LGPL wat betekent dat je de function calls kunt gebruiken en je programma mag linken aan de KDE bibliotheken. Dit telt voor de kdelibs bibliotheken (konqueror en kcontrol etc.) en de koffice bibliotheken (om koffice applicaties te maken). Voor meer informatie, kijk op KDevelop Licensing.


Ben ik bevoegd om commerciële plugins te maken voor KDevelop?

Bij KDevelop zit een libkdeveloplgpl.so bibliotheek om ontwikkeling van commerciële plugins toe te laten. In deze bibliotheek zitten KDevelop interfaces en voorzieningsfuncties onder de LGPL. Als je een professionele licentie van Trolltech hebt kun je je code linken met libkdeveloplgpl en het onder elke licentie maken.


Overige FAQ

Ik ben een applicatie aan het maken die foutafhandeling nodig heeft. Hoe kan ik die maken?

Bewerk het configure.in(.in) bestand in je primaire code map. Hier moet je achter de AC_CHECK_COMPILERS() marco hetvolgende invoeren: 

CXXFLAGS="$CXXFLAGS $USE_EXCEPTIONS"

en hierna moet je je "configure" opnieuw maken door het volgende uit te voeren 

make -f Makefile.dist

in je primaire code map en dan het configurescript opnieuw uitvoeren.


Mijn configure.in.in heeft geen AC_CHECK_COMPILERS() aanroep. Waar moet ik de CXXFLAGS="$CXXFLAGS $USE_EXCEPTIONS" die hierboven is gegeven invoeren?

De nieuwe versies van configure.in.in is versimpeld. Je kan nu de regel toevoegen na de AM_INIT_AUTOMAKE(<pakketnaam>,<versie>).

Waarom zie ik geen compilatie foutmeldingen?

Als je tijdens het compileren merkt dat sommige belangrijke informatie niet gemeld wordt in de Messages Output View scherm, het kan zijn dat het niveau van berichtendetails te laag is. Rechts-klikken in het scherm en selecteer een ander detail niveau van het context menu.


Hoe voeg ik een vertaling toe aan een KDE-programma?

Executeer eerstBuild->Create Messages&Merge en voeg hierna een nieuwe vertaling toe via het Project->Add Translation. Dit creëerd een po/$lang.po bestand. Het $project.pot bestand heeft alle regels in zich.


Hoe voeg ik iconen toe aan een project?

Voeg eerst een nieuw subproject toe via de automake manager genaamd pics. Dan rechts-klik op dat nieuw subproject, kies Settings en dan Prefix. Voeg een nieuwe prefix toe met bijvoorbeeld mijnpics als naam en het pad $(kde_datadir)/$project/pics. Creëer hierna een nieuw doel in het pics subproject en kies Data als Primary, mypics als Prefix en voeg een nepnaam toe. Als laatste voeg je je Iconen toe aan het nieuwe doel. Het laden van het plaatje kan worden gedaan door het invoegen van het kiconloader.h header bestand en door de volgende code dat de pixmap toevoegd met de naam config aan een knop: button->setPixmap(UserIcon("config"))


Hoe kan ik de SVN plugin gebruiken?

Je moet handmatig het project en de code importeren in een subversion repository. Herstart dan KDevelop en een subversion context menu zou moeten verschijnen wanneer je rechts-klikt op bestanden. Intern zoekt de subversion module voor de aanwezigheid van de .svn map in dezelfde map als de file staat waar je op klikt.

Hoe gebruik ik een andere compiler (native of cross compilers)?

Compilers die gezet zijn voor een project:

  • Project->Project Options->Configure Options (alleen autotools gebaseerde projecten, custom projecten hebben zo'n tabblad niet)
  • voeg een nieuwe configurenaam in, b.v. <compilernaam>-<platform>-debug

Tab "General":

  • zet de configure argumenten (b.v. --host en --build opties bij cross-compiling)
  • zet de bouwmap, dit om onduidelijkheid in de configuraties met dezelfde naam te voorkomen
  • zet de vlaggen die nodig zijn (LDFLAGS, ENV)

Tab "C":

  • zet de naam voor je C-compiler binary b.v. CC, gcc, icc, kcc, ... (cross-compilers ook!)
  • zet de vlaggen die nodig zijn (CFLAGS)

Tab "C++":

  • zet de naam voor je C++-compiler binary, b.v. cc, g++, ... (cross-compilers ook!)
  • zet de vlaggen die nodig zijn (CXXFLAGS)

Tab "Fortran":

  • zet de naam voor je Fortran-compiler binary, b.v. g77, g90, gfortran, ... (cross-compilers ook!)
  • zet de vlaggen die nodig zijn (FFLAGS)

Sluit de opties, selecteer je configuratie in Project->Build Configuration, start dan AutoMake & Friends, configure en make.

Bijvoorbeeld, om een cross-compiler voor sparc-sun-solaris2.9 op te zetten:

  • Project -> Project Options -> Configure Options
  • Voeg een nieuwe configuratienaam, b.v. "sparc-debug""
  • Configure Argumenten: 
--host=i686-pc-linux-gnu -- build=sparc-sun-solaris2.9
  • Bouwmap: 
sparc-debug
  • ga naar Tab C, voeg de naam van de cross gcc, b.v. sun-solaris-sparc2.9-gcc, in
  • ga naar Tab C++, voeg de naam van de cross g++, b.v. sun-solaris-sparc2.9-g++, in
  • AutoMake & Friens
  • configure
  • make

Voor cross debuggen zie Op afstand debugging gebruik makend van KDevelop 3.0 in comp.os.linux.embedded newsgroep of Cross_debugging. Het kdevelop.org forum heeft er ook informatie over.

Zorg ervoor dat de binaire bestanden en bibliotheken van de (cross-)compiler beschikbaar zijn in $PATH en $LD_LIBRARY_PATH!

Het kan nodig zijn om sommige autoconf/automake macros in configure.in te veranderen om de compiler volledig te ondersteunen -- bekijk hiervoor het laatste bericht in deze thread op kdevelop.org forum om te kijken wat je moet doen om de gfortran (GNU F90 compiler) werkend te krijgen.


Werken met Clearcase schermen en KDE

Clearcase bestanden zijn normaal opgeslagen onder '/usr/vobs/...' en binaire objecten mogen referenties naar dit pad bevatten. Als jou KDevelop project gedefinieerd was met een ander pad (b.v. '/view/[view name]/usr/vobs'), kan je problemen ondervinden met bestanden die twee keer geopend worden, vreemd breekpuntgedrag en misleidende bestandsbewerking.

Als KDevelop is gestart van een shell in de clearcase scherm zal het nog steeds niet '/usr/vobs/...' zien behalve als KDE zelf was gestart in een clearcase scherm. De volgende aanpak lost deze problemen op:

cp /etc/X11/xinit/xinitrc ~/.xinitrc
Verander de voorlaatste regel van
exec $WINDOWMANAGER
naar
/opt/rational/clearcase/rhat_x86/bin/cleartool setview -exec $WINDOWMANAGER ViewName

Waar ViewName is de naam van het scherm dat je wil gebruiken

Jammergenoeg geeft dit schijnbaar problemen met niet-KDE applicaties (b.v. Firefox) en heeft mijn machine laten crashen. Een ander alternatief is om een andere windowmanager te gebruiken zoals GNOME.
