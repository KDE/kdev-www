Waar kan ik KDevelop krijgen? Hoe installeer ik het?
Het KDevelop Team verspreidt geen binary versies van KDevelop (RPM's). RPM's kunnen daarom worden gevonden op de server van je distrobutie of in de corresponderende subdirectory's van ftp.kde.org of een van haar mirrors
Neem een kijkje op onze download site voor informatie over hoe je moet downloaden en de source code kunt compileren.


Mag ik commerciële applicaties ontwikkelen met KDevelop?
Iedereen en alle entities zijn vrij om KDevelop te gebruiken. De gegenereerde Code kan onder alle licenties die je wilt worden ondergebracht. Daarom heeft de projectgenerator velden om je eigen copyright notice aan de gegenereerde bestanden toe te voegen, maar om het simpeler te maken en om een voorbeeld te hebben is er standaard de GPL licentie. Dus je kan daar je eigen licentie neerzetten.
Als het gaat om het eigenlijke bewerken etc. van je programma is KDevelop net zoals een andere editor, het is een programma dat GPL is en geen implicaties heeft op waar je het voor gebruikt. Dus kan je er commerciële/gesloten software mee schrijven ondanks het feit dat KDevelop zelf GPL is.
Voor het gebruiken van KDE functies of andere vrije software dingen, moet je alleen letten op de licentievoorwaarden op de bibliotheken die je gaat gebruiken. Dus voor commerciële ontwikkeling van KDE/Qt applicaties heb je alleen een professionele licentie nodig van Trolltech. De KDE bibliotheken zijn zelf LGPL wat betekent dat je de function calls kunt gebruiken en je programma mag linken aan de KDE bibliotheken. Dit telt voor de kdelibs bibliotheken (konqueror en kcontrol etc.) en de koffice bibliotheken (om koffice applicaties te maken).
Voor meer informatie, kijk op KDevelop Licensing


Mag ik commerciële plugins voor KDevelop ontwikkelen?
Bij KDevelop zit een libkdeveloplgpl.so bibliotheek om ontwikkeling van commerciële plugins toe te laten. In deze bibliotheek zitten KDevelop interfaces en voorzieningsfuncties onder de LGPL. Als je een professionele licentie van Trolltech hebt kun je je code linken met libkdeveloplgpl en het onder elke licentie maken.


Ik heb KDevelop 3.0 geïnstalleerd maar ik zie alleen een Bestand(File)en een Help menu
Net zoals elke andere KDE applicatie moet je zeker zijn dat KDevelop's plugins worden herkend door KDE. Installeer het in je KDE directory (configure --prefix=/opt/kde3 bijv.) of voeg het KDevelop pad toe aan je KDEDIRS omgevingsvariabele en voer "kbuildsycoca" opnieuw uit. Voorbeeldopdrachtlijn voor bash (ervan uitgaande dat KDevelop is geïnstalleerd in /home/harry/kdevelop):

Hoe kan ik schakelen tussen geopende bestanden met het toetsenbord?
Je kunt ALT-Links en ALT-Rechts gebruiken om naar het vorige/volgende geopende bestand te gaan. Deze toetsen kunnen worden geconfigureerd in de KDevelop setup. Er is ook een Snelle Open (CTRL-SHIFT-O) functie om een bestand te selecteren en openen in een project en Schakel Naar (CTRL-/) functie die het toelaat de eerste letters van een geopend bestand in te typen om ernaar te schakelen.

Hoe kan ik code completie aanzetten voor externe bibliotheken?
Als je een C/C++ project hebt geopend, ga naar het menu Project->Projecteigenschappen(Project Settings), in het Projectopties (Project Options) dialoog kies C++ Specifieke tab (C++ Specific tab), dan Code Completie (Code Completion) tab. Klik op de Add Persistant Class Store knop en vul de New Persistant Class Store wizard in. Als je klaar bent, zorg ervoor dat er de nieuwgemaakte class store is aangevinkt in de lijst van persistente class stores in de Code Completie (Code Completion) tab.

Ik ontwikkel een applicatie waar ik -D opties moet zetten voor de preprocessor.
Als ik deze in KDevelop toevoeg aan mijn projectopties is alles goed, maar als ik mijn pakket verspreid wordt de -D optie, gezet in
KDevelop niet gebruikt. Wat moet ik doen om dit te laten werken?
Bewerk het bestand configure.in(.in) in je bovenste brondirectory. Hier moet je dit
invullen na de AC_CHECK_COMPILERS macro:
CPPFLAGS="$CPPFLAGS -DMIJNDEFINE"
en daarna moet je je nieuwe configure maken door "make -f Makefile.dist" uit te voeren
en daarna het configure script opnieuw uit te voeren.


Is er een andere mogelijkheid dan zelf mijn eigen defines te publiceren?
Een andere mogelijkheid om DEFINE's in je project in te sluiten is
om ze door config.h te publiceren.
Hiervoor moet je de volgende bestanden patchen:
Voeg de volgende lijn toe aan acconfig.h van je bovenste projectdirectory:
#undef MIJNDEFINE

en voeg dan in configure.in(.in) in, na de macro call:
AC_CONFIG_HEADER(config.h):
AC_DEFINE_UNQUOTED(MIJNDEFINE)

(Het zal dus worden gedefinieerd in config.h na je framework te hebben geüpdate met
"make -f Makefile.dist; configure [jouw opties]").

Vereisten voor deze oplossingen zijn:
AC_CONFIG_HEADER(config.h)
in configure.in(.in) en in de sourcecode een
#ifdef HAVE_CONFIG_H
#include <config.h>
#endif
voor de andere includes worden gedaan
(normaal is dit al gedaan in onze projecttemplates).

Een voordeel van deze oplossing is om MIJNDEFINE afhankelijk te makn van
bepaalde situaties (zoals een optie voor configure maken).
Hiervoor moet je natuurlijk een eigen macro functie maken.

Wat is het verschil tussen CPPFLAGS en CXXFLAGS?
CPPFLAGS wordt gebruikt voor defines (dat betekent vlaggen die je alleen nodig hebt om te compileren
CXXFLAGS wordt gebruikt voor compiler switches (bijv. -frtti), die ook van toepassing zijn
op het linking proces.


Ik heb enkele oplossingen gezien zoals het patchen van Makefile.am met Makefile.am with
DEFS+=-DMYDEFINE
of
CXXFLAGS+=-fsigned-char
of
LD_FLAGS+=-L/usr/local/lib
Waarom overweeg je dit niet?
Er zit een bug in automake 1.4, die vlaggen niet toevoegt aan de bestaande.
Met dit commando in Makefile.am, zou je de bestaande instelling van deze vlag verwijderen.
Als je deze vlaggen alleen wilt voor de projectsubdirectory, gebruik:
AM_CPPFLAGS=-DMYDEFINE
of
AM_CXXFLAGS=-fsigned-char
of
AM_LDFLAGS=-L/usr/local/lib

Pas op, deze vlaggen worden alleen gepubliceerd door automake versie 1.4.

Om een van deze vlaggen te zetten voor het hele project, moet je in plaats daarvan
de configure.in(.in) in de bovenste brondirectory patchen.


Hoe kan ik garanderen dat een gebruiker van mijn project het zal bouwen met
automake versie 1.4?
Voeg de string "1.4" toe aan de AUTOMAKE_OPTIONS
variabele in de Makefile.am van de bovenste
brondirectory. Bijv:
AUTOMAKE_OPTIONS = foreign 1.4

Kan AM_CXXFLAGS (binnen een Makefile.am) worden gebruikt voor vlaggen zoals -frtti of
-fexceptions?
Nee! In deze gevallen is er een probleem met de ordering. De
standaard CXXFLAGS variabele gepubliceerd door acinclude.m4.in
bevat al een fno-rtti (-fno_exceptions).
Als je het liet toepassen op AM_CXXFLAGS zou het compiler
commando deze vlaggen invoegen voor de CXXFLAGS, bijv. hier voor -frtti:
gpp ... -frtti ... -fno-exceptions -fno-rtti ......
Dus de laatste vlag (in dit geval nog steeds -fno-rtti) zal worden gebruikt.
KDE_CXXFLAGS zal in plaats daarvan worden ingevoegd ná de standaard
CXXFLAGS, dus het zal op dit lijken:
gpp ..... -fno-exceptions -fno-rtti ...... -frtti...
en voilà het werkt goed.

Pas op: KDE_CXXFLAGS wordt alleen gepubliceerd
door projecten, die am_edit en ascinclude.m4
gebruiken.

Ik gebruik CXXFLAGS="$USE_EXCEPTIONS" binnen configure.in(.in) en het werkt goed,
waarom zou ik jouw oplossing gebruiken?
Als je jouw versie zou gebruiken voor AC_CHECK_COMPILERS() zou een call als
CXXFLAGS="-pedantic" ./configure
niet goed werken.
Als je het gebruikte na AC_CHECK_COMPILERS() zou de
CXXFLAGS gezet in AC_CHECK_COMPILERS() worden gewist.
(Bijvoorbeeld in een "configure --enable-debug" call zou CXXFLAGS worden
veranderd om ook debuginformatie aan je code toe te voegen.)
Gebruik dus aub CXXFLAGS="$CXXFLAGS $USE_EXCEPTIONS".


Ik gebruik additionele headers en bibliotheken voor mijn KDE/QT project.
-I/usr/include/foo in the compiler opties zetten en -L/usr/lib/foo in de linker opties verspreidt de instellingen
niet naar de tarball. Wat moet ik doen?
Dit is het moeilijkste deel om het goed te doen.
Een makkelijke, maar niet de goede oplossing zou zijn om aan configure.in(.in) toe te voegen:

all_includes="$all_includes -I/usr/include/foo"
all_libraries="$all_libraries -L/usr/lib/foo"

[de beste plaats daarvoor is voor de regels:
AC_ARG_PROGRAM
AC_OUTPUT( ..... )
aan het eind van het bestand.]

Problemen hier:
- Wie garandeerd dat de includes altijd zullen zijn in /usr/include/foo...
misschien kan het op sommige systemen in /usr/local/foo/include zijn
(en misschien zijn de bibliotheken daar in /usr/local/foo/lib)??
- Misschien heb ik de foo-dingen niet geïnstalleerd zodat het
bouwen faalt, maar zou het beter zijn als configure al zou falen met
een handiger bericht.

Dus om het goed te doen zou je een macro functie toe moeten voegen aan
configure.in(.in), die bepaalde testen en een zoektocht doet voor een geïnstalleerde "foo".

Als je dit wilt doen, kunnen we je hier alleen een hint geven... probeer
de acinclude.m4.in te begrijpen of maak een KDE-normal-ogl project,
daar vind je een bestand qgl.m4, die zal worden toegevoegd aan acinclude.m4
(zie "Makefile.dist"). Dit laat een uitgebreid voorbeeld zien om te zoeken
naar QT-OpenGL-Lib, je moet het herschrijven voor jouw probleem.

Misschien heb je geluk en vind je al een oplossing voor je probleem in
acinclude.m4.in, zodat je een al gemaakte functie kunt gebruiken.

Hoe maak ik een simpel project met alleen een Makefile (ook wel een Custom Project genoemd) ?
Create a Custom Project. There is no direct way to create a custom project (i.e. a project which does use its own makefiles). Use Project->Import Existing Project instead. Remember to set the appropriate Project Type, labeled by an additional (Custom Makefiles) in the dialog. Use Project Options Early. Whenever you start a new project do not forget to set the Project->Project Options... to your needs.

Waarom zie ik de compilatiefoutmeldingen niet?
Als het je opvalt dat tijdens compilaties waardevolle informatie mist in het Messages Output View venster, kan het zijn dat het nivo van berichtdetail te laag is gezet. Rechtsklik in het venster en selecteer een ander detaillevel uit het contextmenu.

Hoe maak ik Nieuwe Bestanden in een Project?
Selecteer het subproject in de bovenste helft van de Automake Manager, klik dan rechts in de onderste helft op de titel van de groep waar je je bestanden aan toegevoegd wil hebben en selecteer Create New File....

Hoe voeg ik Bestaande Bestanden toe aan een project?
Selecteer het subproject in de bovenste helft van de Automake Manager, klik dan rechts in de onderste helft op de titel van de groep waar je je bestanden aan toegevoegd wil hebben en selecteer Add Existing Files....

Hoe verwijder ik een Bestand uit een Project?
Selecteer het subproject in de bovenste helft van de Automake Manager, klik dan rechts in de onderste helft bestand in de groep waar je het van verwijderd wil hebben, en selecteer Remove.

Wat moet ik doen als het project niet meer bouwt na te hebben geschakeld van/naar standaarddoel?
Er is een probleem met de automake/autoconf machinerie. Als Project->Build Configuration drie bouwdirectory's laat zien om van te kiezen: default, optimized en debug, blijf altijd of bij de default of de debug/optimized doelen.
Als je je project met default geconfigureerd hebt bouwt het niet meer met debug of optimized.
Als je je project met debug of optimized geconfigureerd hebt bouwt het niet meer met default.


"Verkeerde autoconf versie" ("Wrong autoconf version") etc. fout.
Er zijn enkele foutmeldingen die gaan over te oude versies van autoconf etc., waardoor configure niet goed werkt. Voer autoreconf in de directory tree uit waar de betrokken configure.in bestanden zijn. Dit commando zal proberen de informatie in de GNU Build System bestanden te updaten. Zie man autoreconf voor meer informatie.
