Unterhalb des KDevelop-Systems liegt das GNU Autoconf System. Autoconf ist eine Reihe von Werkzeugen, die Makefiles und Konfigurationsskripe erstellen. Es ist mächtig, flexibel und fast komplett unverständlich, besonders wenn es die Aufgabe ist, eine einfache Abhängigkeit zu einem Programm hinzuzufügen.

Das KDE admin/ Verzeichnis enthält einige Konfigurationsdateien und -skripte, die diesen Prozess vereinfachen sollen, mit wenig Erfolg. Die wichtigsten Befehle, um die KDevelop Autoconf Einstellungen zu verändern, sind make -f admin/Makefile.common (und das verehrte rm config.guess wenn Sie noch KDevelop 2.x benutzen). Der Aufruf von make mit dem administrativen Makefile generiert die Autoconf-Einstellungen und Löschen von config.guess leert den Konfigurations-Cache, wie es bei jeder etwaigen festen Einstellung wäre.

Ein bekanntes Szenario, das die Modifikation der Autoconf-Dateien erfordert, ist, eine externe Abhängigkeit zu einer Shared Library, die nicht Teil der KDevelop GUI ist, hinzuzufügen; z.B. libxml2. Es ist einfach, die Befehle `xml2-config --cflags` und `xml2-config --libs` zur CPPFLAGS- und LDFLAGS-Sektion der KDevelop-Optionen für das verwendete Profil hinzuzufügen. Wie auch immer, diese Werte werden nur benutzt, um in KDevelop zu kompilieren und gehen verloren, wenn man die Quellen abpackt (.tar.gz).

Diese Abhängigkeit dann hinzuzufügen ist die Aufgabe von Autoconf. Zwei Dateien sind in diesem Szenario zu modifizieren: configure.in.in und Makefile.am. configure.in.in ist eine von KDevelop beim Erstellen eines neuen Projekts erstellte Vorlage. KDevelop überschreibt Änderungen an dieser Datei nicht. Makefile.am ist auch eine Vorlage, aber KDevelop 2.x könnte Teile der Datei überschreiben. Die Sektionen, die überschrieben werden könnten, sind klar markiert und die Änderungen außerhalb dieser Sektionen werden sicher beibehalten. Diese Einschränkung ist mit KDevelop 3.0 verschwunden.

Also was bewirken diese Dateien?

configure.in.in ist eine von m4, der GNU Makro Processing Sprache, verarbeitete Datei. Dieselbe Makrosprache wird benutzt, um die Konfigurationsdateien von Sendmail zu erstellen. Obwohl sie eine lernenswert Sprache ist, werden wir sie hier nicht erklären. Stattdessen ist unten ein Codeteil aufgelistet, den man in configure.in.in einfügen kann, um auf libxml2 zu prüfen:

dnl -----------------------------------------------------------------------
dnl libxml2 check from http://websvn.kde.org/*checkout*/branches/KDE/3.5/kdelibs/kdoctools/configure.in.in

KDE_FIND_PATH(xml2-config, XML_CONFIG, [${prefix}/bin ${exec_prefix}/bin /usr/local/bin /opt/local/bin], [
  AC_MSG_WARN([Could not find libxml2 anywhere, check ftp://xmlsoft.org/ for libxml >= 2.4.8.
  HELP_SUBDIR=
])

if test -n "$XML_CONFIG"; then
  vers=`$XML_CONFIG --version 2>/dev/null | sed -e 's/libxml //' | awk 'BEGIN { FS = "."; } { printf "%d", ($1 * 1000 + $2) * 1000 + $3;}'`
  if test -n "$vers" && test "$vers" -ge 2005000
  then
     LIBXML_LIBS="`$XML_CONFIG --libs`"
     LIBXML_RPATH=
     for args in $LIBXML_LIBS; do
          case $args in
            -L*)
               LIBXML_RPATH="$LIBXML_RPATH $args"
               ;;
          esac
     done
     LIBXML_RPATH=`echo $LIBXML_RPATH | sed -e "s/-L/-R/g"`
     LIBXML_CFLAGS="`$XML_CONFIG --cflags`"
  else
     AC_MSG_WARN([You need at least libxml 2.5.0])
     HELP_SUBDIR=
  fi
fi

AC_SUBST(LIBXML_LIBS)
AC_SUBST(LIBXML_CFLAGS)
dnl -----------------------------------------------------------------------

Dieses Code-Segment sucht einfach nach dem xml2-config Programm in einigen vordefinierten Pfaden. Der Pfad zur Ausführbaren ist in der Variable XML_CONFIG, dem zweiten Parameter, gespeichert. Wenn das Programm nicht gefunden werden kann, bleibt die Variable leer und eine Fehlermeldung wird ausgedruckt.

AC_SUBST(LIBXML_LIBS)

Und mache die Variablen LIBXML_LIBS und LIBXML_CFLAGS für den Rest des Systems zugänglich. LIBXML_RPATH wird nicht benutzt, daher ist es nicht hier aufgeführt, aber verfügbar für den Export (export).

Jetzt, da diese Variablen zum Rest von Autoconf "exportiert" sind, ist es Zeit, sie in die Makefiles einzufügen. Glücklicherweise ist das eine viel einfachere Aufgabe.

Im Unterverzeichnis, das die Dateien zum Zugriff auf Headers und Bibliothekten enthält (das sollte das "project/" Unterverzeichnis in project sein), ist eine Datei namens Makefile.am. Ersetze diese Zeilen: 

# set the include path found by configure
INCLUDES= $(all_includes)

# the library search path.
myAppName_LDFLAGS = $(all_libraries)
durch diese: 
# set the include path found by configure
INCLUDES= $(all_includes) $(LIBXML_CFLAGS)

# the library search path.
myAppName_LDFLAGS = $(all_libraries) $(LIBXML_LIBS)

Zu beachten ist wieder das Segment, das KDevelop überschreiben wird, wenn Sie Version 2.x benutzen. Wenn Sie Version 3 benutzen, können Sie diese Optionen entweder im Automake-Manager oder von Hand setzen (der Automake-Manager wird sie aufnehmen, wenn das Projekt neu geladen wird).

Der letzte notwendige Schritt zur Integration besteht einfach darin, make aufzurufen. Abhängig von der Komplexität der Abhängigkeiten kann es trotzdem nötig sein, make -f admin/Makefile.common aufzurufen, was dafür sorgt, dass Autoconf seine Dateien erneuert und aufgeräumt wird, wo es KDevelop für nötig hält. Wenn zum Beispiel die Versionprüfungen verändert wurden, kann es notwendig sein, die Datei config.cache zu entfernen und configure nochmal aufzurufen. Ab hier ist make wirklich der letzte Schritt.

Gratulation! Sie haben nun eine funktionierende libxml2-Prüfung und ein Softwarepaket, das damit innerhalb und außerhalb von KDevelop kompiliert werden kann!
