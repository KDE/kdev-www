Woher bekomme ich KDevelop? Wie installiere ich es?
Das KDevelop Team veröffentlicht keine Binärversionen von KDevelop (RPMs). RPMs befinden sich auf dem Server Ihres Distributors oder in den entsprechenden Unterverzeichnissen von ftp.kde.org oder einem seiner Spiegelserver
Bitte schauen Sie sich unsere Downloadseite für Informationen zum Download und Compilieren an.


Darf ich kommerzielle Anwendungen mit KDevelop entwickeln?
Everyone and all entities are free to use KDevelop. The generated Code can be put under all licenses you wish. Therefore, the project generator has the fields for adding your own copyright notice to the generated files, but to make things simpler and to have an example, the default is a GPL license notice. So you can add your license notice there.
Beim Editieren von Programmen verhält sich KDevelop wie jeder andere Editor, das Programm an sich steht unter der GPL und es gibt keine weiteren Einschränkungen was Sie damit mit machen. Sie können kommerzielle/closed source Software mit schreiben und die Tatsache, dass KDevelop unter der GPL steht vernachlässigen.
Falls Sie KDE Funktionen und andere Teile von freie Software verwenden, beachten Sie bitte die Lizenzbestimmungen der von Ihnen verwendeten Bibliotheken. Für die kommerzielle Entwicklung von KDE/Qt Anwendungen benötigen Sie eine Lizenz von Trolltech. Die KDE Bibliotheken selbst sind LGPL, d.h. Sie können die Funktionsaufrufe nutzen und Ihre Anwendung mit der Bibliothek verlinken. Das gilt für die kdelibs Bibliotheken, die kdebase Bibliotheken (konqueror, kcontrol etc.) und koffice Bibliotheken (um KOffice Anwendungen zu erstellen).
Für weitere Informationen besuchen Sie bitte http://kdevelop.org/1.3/doc/programming/index-16.html


Darf ich kommerzielle Plugins für KDevelop entwickeln?
KDevelop enthält eine libkdeveloplgpl.so Bibliothek um die Entwicklung von kommerziellen Plugins zu ermöglichen. Diese unter der LGPL veröffentlichte Bibliothek enthält KDevelop Schnittstellen und Hilfsfunktionen. Falls Sie eine kommerzielle Lizenz von Trolltech haben können Sie Ihren Code mit libkdeveloplgpl verlinken und unter jeder beliebigen Lizenz veröffentlichen.


Ich habe KDevelop 3.0 compiliert und installiert, aber ich sehe nur das Datei- und das Hilfe-Menü
Wie bei jeder anderen KDE Anwendung müssen Sie sicherstellen, dass KDevelop's Plugins von KDE erkannt werden. Entweder installieren Sie sie in Ihr KDE Verzeichnis (configure --prefix=/opt/kde3 beispielsweise) oder fügen Sie den KDevelop Pfad zu Ihrer KDEDIRS Umgebungsvariablen hinzu und führen "kbuildsycoca". Beispielshafte Kommandozeilen für Bash sind (unter der Annahme, dass KDevelop in /home/harry/kdevelop installiert ist):
export KDEDIRS=/home/harry/kdevelop:$KDEDIRS && kbuildsycoca


Wie kann ich mit der Tastatur zwischen den offenen Dateien wechseln?
Sie K&oum;nnen ALT-Links und ALT-Rechts benutzen um zur nächsten offenen Datei zu gelangen. Diese Tasten können in den KDevelop Einstellungen geändert werden. Es gibt auch eine Schnell Öffnen Funktion (STRG-UMSCHALT-O) um eine Datei im Projekt auszuwählen und zu öffnen und eine Wechseln Zu Funktion (STRG-/), die es ermöglicht den ersten Buchstaben einer offenen Datei anzugeben und zu dieser zu wechseln.

Wie kann ich die Codevervollständigunge für externe Bibliotheken aktivieren?
When you have any C/C++ project opened, go to menu Project->Project Settings, in Project Options dialog choose C++ Specific tab, then Code Completion tab. Push Add Persistant Class Store button and complete the New Persistant Class Store wizard. After completing the wizard, make sure that the newly created class store is checked in the list of persistant class stores at Code Completion tab.

Ich bin Entwickler einer Anwendung in der ich dem Präprozessor -D Optionen übergeben muss. Wenn ich diese in KDevelop zu meinen Projektoptionen hinzufüge funktioniert alles, aber wenn ich mein Paket veröffentliche werden die -D Optionen, die ich in KDevelop gesetzt habe nicht benutzt. Was muss ich tun damit das funktioniert?
Editieren Sie die Datei configure.in(.in) in Ihrem Quellcodeverzeichnis. Tragen Sie nach dem AC_CHECK_COMPILERS Makro folgendes ein:
CPPFLAGS="$CPPFLAGS -DMYDEFINE"
und erzeugen Sie eine neue "configure" Datei indem Sie "make -f Makefile.dist" eingeben. Danach müssen Sie das configure Script nochmals aufrufen.


Gibt es einen anderen Weg meine eigenen defines zu verwenden?
Eine andere Möglichkeit DEFINEs in Ihrem Projekt einzubinden ist Sie in die Datei config.h zu schreiben.
Dafür müssen Sie die folgenden Zeilen bearbeiten:
Fügen Sie zur acconfig.h in Ihrem Projektverzeichnis die folgende Zeile hinzu:
#undef MYDEFINE

und fügen Sie in die configure.in(.in), nach dem Makroaufruf AC_CONFIG_HEADER(config.h) folgendes ein:
AC_DEFINE_UNQUOTED(MYDEFINE)

(So wird es in config.h definiert nachdem Sie Ihr Framework mit
"make -f Makefile.dist; configure [Ihre Optionen]" updaten.

Voraussetzungen für diese Lösung sind:
AC_CONFIG_HEADER(config.h)
in der configure.in(.in) und im Sourcecode ein
#ifdef HAVE_CONFIG_H
#include <config.h>
#endif
bevor irgendwelchen anderen Dateien eingebunden werden (normalerweise ist das in unseren Projekt Templates bereits für Sie erledigt.

Übrigens: Ein Vorteil dieser Lösung ist, dass MYDEFINE von bestimmten Situationen abhängen kann (z.B. von einer configure Option). Dafür müssen Sie natürlich eine eigene Makrofunktion schreiben.


Ich entwickle eine Anwendung die exception handling benötigt. Wie kann ich das fertigstellen?
Bearbeiten Sie die Datei configure.in(.in) in im Wurzelverzeichnis Ihres Quelltexts. Fügen Sie nach dem AC_CHECK_COMPILERS() Makro folgendes ein:
CXXFLAGS="$CXXFLAGS $USE_EXCEPTIONS"
und ereugen Sie eine neue configure-Datei indem Sie "make -f Makefile.dist" im Wurzelverzeichnis Ihres Quelltexts aufrufen und danach das configure Script neu starten.


Meine configure.in.in hat keine AC_CHECK_COMPILERS() Aufrufe. Wo soll ich die oben erwähnten CXXFLAGS="$CXXFLAGS $USE_EXCEPTIONS" hinschreiben?
In neueren Versionen von configure.in.in wurde das ganze vereinfacht, Sie können die Zeile jetzt nach AM_INIT_AUTOMAKE(<package-name>,<version>) hinzufügen.


Ich entwickle eine Anwendung die run-time-type-info eincompiliert benötigt. Wie kann ich das bewerkstelligen?
Bearbeiten Sie die Datei configure.in(.in) im Wurzelverzeichnis Ihres Quellcodes. Dort fügen Sie nach dem AC_CHECK_COMPILERS Makro folgendes ein:
CXXFLAGS="$CXXFLAGS $USE_RTTI"
und nachdem Sie eine neue configure-Datei mit dem Befehl "make -f Makefile.dist" in Ihrem Quellcodeverzeichnis erstellt und Ihr configure Script neu gestartet haben.

Was ist der Unterschied zwischen CPPFLAGS und CXXFLAGS?
CPPFLAGS wird für defines benutzt (d.h. Flags die nur für die Compilierung interessant sind).
CXXFLAGS wird für Compiler Schalter (z.B. -frtti) benutzt, die auch beim Linking eine Rolle spielen.


Ich habe einige Lösungen Makefile.am mit
DEFS+=-DMYDEFINE
oder
CXXFLAGS+=-fsigned-char
oder
LD_FLAGS+=-L/usr/local/lib
zu patchen. Warum machen Sie das nicht?
Es gibt einen Bug in automake 1.4, der Flags nicht zu bereits bestehenden Flags hinzufügt. Mit diesem Kommando in Makefile.am würden Sie bereits bestehende Einstellungen des jeweiligen Flags löschen.
Falls Sie ein Flag nur für ein Unterverzeichnis benötigen, nutzen Sie bitte die folgenden Befehle:
AM_CPPFLAGS=-DMYDEFINE
oder
AM_CXXFLAGS=-fsigned-char
oder
AM_LDFLAGS=-L/usr/local/lib

Bitte beachten Sie, dass diese Flags nur von automake 1.4 veröffentlicht werden.

Um eines dieser Flags für das komplette Projekt zu verwenden, bearbeiten Sie bitte die configure.in(.in) im Wurzelverzeichnis Ihres Quelltexts.


Wie kann ich sicherstellen, dass mein Projekt mit automake 1.4 erzeugt wird?
Fügen Sie zur Variable AUTOMAKE_OPTIONS in Makefile.am im Wurzelverzeichnis Ihres Quellcodes die Zeichnkette "1.4" hinzu. Beispielsweise
AUTOMAKE_OPTIONS = foreign 1.4


Ich habe ein existierendes Unterprojekt zu meinem Projekt hinzugefügt und jetzt compiliert es nicht mehr.
Das kann passieren wenn das Erzeugungsverzeichnis nicht das oberste Quellcodeverzeichnis ist. In diesem Fall führen Sie bitte "make distclean" aus bevor Sie das Unterprojekt hinzufügen.


Kann AM_CXXFLAGS (in Makefile.am) für die Flags wie -frtti oder -fexceptions benutzt werden?
Nein! In diesen Fällen besteht ein Problem mit der Reihenfolge. Die Standard CXXFLAGS Variable veröffentlicht von acinclude.m4 enthält bereits -fno-rtti (-fno-exceptions).
Wenn Sie AM_CXXFLAGS verwenden würden, würde das Compiler Kommando diese vor CXXFLAGS einfügen, z.B. hier für -frtti:
gpp ... -frtti ... -fno-exceptions -fno-ftti ...
Damit würde das letzte Flag (in diesem Fall weiterhin -fno-ftti) weiterhin benutzt werden.
KDE_CXXFLAGS stattdessen wird nach dem normalen CXXFLAGS eingefügt, sodass es im Endeffekt so ausschaut:
gpp ... -fno-exceptions -fno-rtti ... -frtti ...
und alles funktioniert perfekt.
Achtung: KDE_CXXFLAGS wird nur von Projekten veröffentlicht die am_edit und acinclude.m4 verwenden.

Ich benutze CXXFLAGS="$USE_EXCEPTIONS" in der configure.in(.in) und es funktioniert perfekt. Warum sollte ich eure Lösung benutzen?
Wenn Sie Ihre Version vor AC_CHECK_COMPILERS() verwenden, ein Aufruf wie
CXXFLAGS="-pedantic" ./configure
würde nicht korrekt funktionieren.
Falls Sie es nach AC_CHECK_COMPILERS() verwenden, die CXXFLAGS in AC_CHECK_COMPILERS() würden gelöscht werden.
(Beispielsweise in einem "configure --enable-debug" Aufruf würde CXXFLAGS dahingehend geändert, Debug Information einzufügen).
Aus all diesen Gründen verwenden Sie stattdessen besser CXXFLAGS="$CXXFLAGS $USE_EXCEPTIONS".


Ich benutze zusätzliche Header und Bibliotheken für mein KDE/Qt Projekt. Wenn ich -I/usr/include/foo in den Compiler Optionen und -L/usr/lib/foo in den Linker Optionen setze, werden diese Einstellungen nicht im Tarball übernommen. Was muss ich tun?
Dies ist ein Problem das sich nicht so leicht richtig lösen läßt.
Ein einfacher, aber falscher Weg würde sein in der configure.in(.in) folgendes hinzuzufügen:

all_includes="$all_includes -I/usr/include/foo"
all_libraries="$all_libraries -L/usr/lib/foo"

[der beste Platz für das ist vor diesen Zeilen:
AC_ARG_PROGRAM
AC_OUTPUT( ..... )
am Ende der Datei.]

Probleme mit dieser Lösung:
- Wer kann sicherstellen dass die includes immer in /usr/include/foo sind? Vielleicht sind sie in einigen Systemen unter /usr/local/foo/include (und vielleicht sind die benötigten Bibliotheken in diesem Fall unter /usr/local/foo/lib)?
- Vielleicht hat der Benutzer das foo-Zeugs nicht installiert und die Erzeugung würde fehlschlagen, aber es wäre besser, das bereits in configure mit einer sinnvollen Meldung bekanntzugeben.

Der richtige Weg ist eine Makrofunktion in configure.in(.in) hinzuzufügen, die einige Tests und Suchen für "foo" ausf&uum;hrt.

Falls Sie das tun wollen, gibt's hier nur einen kleinen Hinweis... versuchen Sie die acinclude.m4.in zu verstehen oder erzeugen Sie ein KDE-normal-ogl Projekt, dort finden Sie eine Datei qgl.m4, die zu acinclude.m4 umgewandelt wird (siehe "Makefile.dist"). Diese Datei enthält ein ausführliches Beispiel für die Suche nach Qt-OpenGL-Lib, das Sie natürlich noch Ihren Bedürfnissen anpasssen müssen.

Vielleicht haben Sie auch Glück und finden bereits eine Lösung für Ihr Problem in der acinclude.m4.in, sodass Sie eine bereits fertige Funktion verwenden können.
