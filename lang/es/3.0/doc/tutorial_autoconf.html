Remarco que el sistema KDevelop es el sistema GNU autoconf. autoconf es un conjunto de herramientas que genera programas Makefiles y scripts configure. Es muy potente, flexible, y casi completamente incomprensible, especialmente cuando la tarea es añadir una simple dependencia a un programa.

El directorio admin/ de KDE contiene algunos ficheros de configuración y scripts que intentan simplificar este proceso, con un grado limitado de éxito. Los comandos más importante que use cuando modifique las configuraciones de autoconf de KDevelop son make -f admin/Makefile.common(y el venerable rm config.guess si aun está usando KDevelop 2.x). Llamando a make con el Makefile administrativo regenerará las configuraciones de autoconf, borrará el config.guess y limpiará la cache de confié, como este, para cualquiera de las configuraciones potencialmente caducas.

Un escenario normal el cual requiere modificar estos ficheros de autoconf es añadir una dependencia exterior a una biblioteca compartida que no es parte de la GUI de KDevelop, como libxml2. Es fácil añadir los comandos `xml2-config --cflags` y `xml2-config --libs` a la sección CPPFLAGS y LDFLAGS de las opciones de KDevelop para el perfil con el que estés compilando; sin embargo, estos valores sólo se utilizan cuando está compilando dentro de KDevelop y se pierde cuando se crea la distribución del fuente (.tar.gz).

Añadir esta dependencia, entonces, es una tarea de autoconf. Hay dos ficheros que se modificar en este escenario: configure.in.in y Makefile.am. configure.in.in es una plantilla creada por KDevelop cuando se crea por primera vez el proyecto. KDevelop no sobrescribe ninguno de los cambios realizados sobre el este fichero. Makefile.am es también una plantilla, pero KDevelop 2.x puede sobrescribir partes del fichero. Las secciones que serán sobrescritas están claramente marcadas, y los cambios realizados fuera de esta sección están protegidos de forma segura. Esta limitación se ha quitado en KDevelop 3.x

¿Así que qué podemos hacer con estos ficheros?

configure.in.in es un fichero que se parsee con m4, el lenguaje de procesado de macro de GNU. Este es el mismo lenguaje de macro que se usa para generar los ficheros de configuración para sendmail, así como para otras muchas cosas. Aunque sea una cosa que vale la pena aprender, no se explicará aquí. En vez de eso, lo listado más abajo es un recorte de código que se puede pegar dentro de configure.in.in para activar una comprobación para libxml2:

dnl -----------------------------------------------------------------------
dnl libxml2 check from http://websvn.kde.org/*checkout*/branches/KDE/3.5/kdelibs/kdoctools/configure.in.in

KDE_FIND_PATH(xml2-config, XML_CONFIG, [${prefix}/bin ${exec_prefix}/bin /usr/local/bin /opt/local/bin], [
  AC_MSG_WARN([Could not find libxml2 anywhere, check ftp://xmlsoft.org/ for libxml >= 2.4.8.
  HELP_SUBDIR=
])

if test -n "$XML_CONFIG"; then
  vers=`$XML_CONFIG --version 2>/dev/null | sed -e 's/libxml //' | awk 'BEGIN { FS = "."; } { printf "%d", ($1 * 1000 + $2) * 1000 + $3;}'`
  if test -n "$vers" && test "$vers" -ge 2005000
  then
     LIBXML_LIBS="`$XML_CONFIG --libs`"
     LIBXML_RPATH=
     for args in $LIBXML_LIBS; do
          case $args in
            -L*)
               LIBXML_RPATH="$LIBXML_RPATH $args"
               ;;
          esac
     done
     LIBXML_RPATH=`echo $LIBXML_RPATH | sed -e "s/-L/-R/g"`
     LIBXML_CFLAGS="`$XML_CONFIG --cflags`"
  else
     AC_MSG_WARN([You need at least libxml 2.5.0])
     HELP_SUBDIR=
  fi
fi

AC_SUBST(LIBXML_LIBS)
AC_SUBST(LIBXML_CFLAGS)
dnl -----------------------------------------------------------------------

  if test -n "$vers" && test "$vers" -ge 2005000
  then

Si la variable XML_CONFIG esta asignada, especificando la ruta correcta, entonces pregunta por la versión. Este programa requiere una versión 2.5.0 o superior, así que comprueba este valor aquí.

Si la correspondencia de versión fue correcta, ciertas variables se definirán para que se puedan usar por otras partes del sistema autoconf. Estas variables serán LIBXML_LIBS, LIBXML_RPATH, y LIBXML_CFLAGS.

  else
     AC_MSG_WARN([You need at least libxml 2.5.0])
     HELP_SUBDIR=
  fi

Si se encuentra libxml2, pero la versión es demasiado antigua, imprime un error aquí.

fi

AC_SUBST(LIBXML_LIBS)
AC_SUBST(LIBXML_CFLAGS)

Y finalmente, hace las variables LIBXML_LIBS y LIBXML_CFLAGS accesibles al resto del sistema. LIBXML_RPATH no se utilizará, así que no se lista aquí, pero esta disponible para exportar.

Ahora que estas variables se han "exportado" al resto del autoconf, es el momento de insertarlas dentro del Makefiles. Afortunadamente, esto es una tarea mucho más sencilla.

Desde el subdirectorio que contiene los ficheros que necesita para acceder a estas cabeceras y bibliotecas (este puede ser el subdirectorio del proyecto/ dentro del proyecto, por ejemplo), hay un fichero llamado Makefile.am. Reemplaza estas líneas: 

# asigna la ruta del include encontrado por configure
INCLUDES= $(all_includes)

# la ruta de búsqueda de la biblioteca.
myAppName_LDFLAGS = $(all_libraries)
con esto: 
# asigne la ruta del include encontrado por configure
INCLUDES= $(all_includes) $(LIBXML_CFLAGS)

# la ruta de búsqueda de la biblioteca.
myAppName_LDFLAGS = $(all_libraries) $(LIBXML_LIBS)

De nuevo, tome nota del segmento que KDevelop sobrescribirá si usa la versión 2.x. Si está usando la versión 3, puede asignar estas opciones con el administrador de automake o a mano (el administrador de automake los cogerá cuando recoge el proyecto).

El paso final requiere para completar esta integración simplemente debe ejecutar make. Dependiendo de la complejidad de las dependencias, sin embargo, puede que sea necesario ejecutar make -f admin/Makefile.common, lo que fuerza a autoconf a refrescar sus ficheros y hacer cualquier otra gestión interna que KDevelop piense que es importante. Si, por ejemplo, las comprobaciones de versión han cambiado, también puede ser necesario borrar el fichero config.cache y reejecutar configure. En este punto, make verdaderamente es el último paso.

¡Enhorabuena! Ahora tiene un comprobador libxml2 funcionando y un paquete de software que puede ser compilado dentro y fuera de KDevelop!
