Работа с файлами

Где находится менеджер Automake в KDevelop?

Откройте вкладку в правой части окна KDevelop.


Как создавать новые файлы в проекте?

Выберите подпроект в верхней части области менеджера Automake, затем щлкните правой кнопкой мыши в нижней части на названии группы, в которую требуется добавить файлы и выберите Создать новый Файл...

Как удалять файлы из проекта?

Выберите подпроект в верхней части области менеджера Automake, затем в нижней части окна откройте список файлов группы с ненужным файлом, затем щлкните правой кнопкой мыши на файле и выберите Удалить...

Как добавить существующие файлы в проект?

Способ добавления в проект уже существующих файлов зависит используемой системы сборки, но в любом случае добавляемые файлы необходимо предварительно поместить в каталог src/ проекта KDevelop.

Затем:

  • Если проект использует Automake (смотрите: FAQ#Где находится менеджер Automake в KDevelop?), то добавить существующие файлы можно просто выбрав подпроект в верхней части менеджера Automake и группу, в которую нужно добавить файлы, в нижней, а затем щёлкнув правой кнопкой мыши и выбрав в контекстном меню "Добавить существующие файлы..."
  • Если проект использует Qmake, действия по добавлению файлов в проект аналогичны действиям для Automake.
  • Если для сборки проекта используются файлы Makefile, создаваемые вручную, то для добавления файлов в проект требуется самостоятельная правка файлов Makefile.


Работа с проектами

Как конвертировать мой VC++ проект?

KDevelop не предоставляет возможности автоматического импорта проектов VC++. Однако, вы можете конвертировать проект, создав новый проект KDevelop на основе шаблонов Мастера новых проектов, таких как шаблон C++->Простая программа Hello world. Затем скопируйте исходные файлы (*.cpp и *.h) из проекта VC++ в папку src/ нового проекта Automake. Наконец, проследуйте приведенным выше инструкциям по добавлению существующих файлов в проект.

Частичным решением проблемы может стать использование скрипта dsw2make. Ему передатся на вход файл проекта VC++ (*.dsw) и сопутствующие ему файлы dsp, после чего он создат Makefile для каждого из них. Хотя это работает, могут возникать некоторые несоответствия, обычно связанные с путями в файловых системах, регистром и концами строк (CR/LF). Для решение последней прооблемы используйте приложения наподобие dos2unix или d2u. Теперь вы должны иметь обычный проект, основанный на наборе файлов Makefile. Импортируйте проект, как уже существующий проект.

Как создать полностью пустой проект?

  • Выберите Проект->Новый проект...->C++->Простая программа Hello world и далее в мастере
  • Удалите из проекта единственную цель сборки
  • Отметьте Удалить также исходные файлы в появившемся диалоге

Как создать простой проект с файлами Makefile, сопровождающимися вручную (Custom Makefiles Project)?

Создание простого проекта. Нет прямого пути, чтобы создать простой проект (использующий собственные Makefile). Вместо этого используйте Проект->Импортировать существующий проект. Не забудьте выставить соответствующий Тип проекта "Custom Makefiles" в диалоге. Всегда, создавая новый проект, не забывайте делать ревизию настроек в Проект->Параметры проекта...


Как создать простой Automake проект с иерархией каталогов?

В проекте с иерархией каталогов каждая папка содержит группы cpp/h файлов, которые должны быть собраны в один исполняемый двоичный файл. 


Пример иерархии:
  src
  src/gui
  src/gui/components
  src/logic
  src/logic/algorithms
  src/logic/sublogic

Это можно сделать, используя вспомогательные библиотеки. Файлы каждой поддиректории будут компилироваться и компоноваться в архивы libtool, после чего они будут компоноваться в единый исполняемый файл и связываться с используемыми общими библиотеками.

  • Шаг 1:
    • В менеджере Automake добавьте подпроект для каждой подпапки:
    • Automake Manager -> [щелчок правой кнопкой мыши на папке] -> Добавить подпроект
  • Шаг 2:
    • Добавьте цель Libtool в каждом подпроекте и свяжите е с .cpp файлами в соответствующем подкаталоге:
      • Automake Manager -> [щелчок правой кнопкой мыши на папке] -> Добавить цель...
      • Первичная: Библиотека Libtool
      • Префикс: noinst
      • Имя файла: <имя файла>.la
  • Шаг 3:
    • Добавьте архивы в исполняемый файл
      • Automake Manager -> [щелчок правой кнопкой мыши на исполняемом файле] -> Параметры -> Библиотеки
  • Шаг 4:
    • Добавьте зависимости для каждого подпроекта
    • Если "labalgo.la" зависит от классов в "libbase.la":
      • Automake Manager -> [щелчок правой кнопкой мыши на libalgo] -> Параметры -> Библиотеки (отметьте флажок рядом с libbase.la) 
Пример:
 src/logic/base [libbase.la]
 src/logic/algorithm [libalgo.la]

Примечание: Если вы забыли указать зависимости, добавьте их зависимости, выполните make clean для подпроекта и перекомпилируйте весь проект (F8).


Для чего предназначены "конфигурации сборки" ("build configurations")?

  • debug:
    • для объектных и исполняемых файлов создатся отдельная папка debug
    • ключи компилятора -g3 -O0 включают отладочную информацию в объектные файлы и отключают оптимизацию
  • optimized:
    • для объектных и исполняемых файлов создатся отдельная папка "optimized"
    • Ключи -O2 -g0 включают оптимизацию и выключают добавление отладочной информации
  • default:
    • Не устанавливаются никакие дополнительные ключи компилятора и каталоги сборки
    • Библиотеки и бинарные файлы будут находится среди исходных файлов
    • Важно! Не используйте "default"(конфигурацию по умолчанию) одновременно с "debug" и "optimized". "debug" и "optimized" могут использоваться совместно без проблем, так как для них используются различные папки сборки.

Что делать, если проект не собирается повторно после переключения на/от цели по умолчанию?

Это проблема связана с automake/autoconf. Если Проект->Конфигурация сборки позволяет выбрать все три варианта: default, optimized и debug, используйте всегда либо default, либо debug/optimized.

  • После выбора конфигурации default конфигурации debug и optimized перестают быть доступными.
  • После выбора конфигурации debug или optimized конфигурация default становится недоступной.


Как добавить подпроект?

Выберите automake-менеджер. В верхней части находятся проекты. Чтобы добавить подпроект, выберите проект, куда его следует добавить, затем щлкните на значке "Добавить подпроект". В файловой системе и менеджере Automake будет создана новая папка с именем подпроекта.

Теперь вам нужно создать цель libtool для нового подпроекта. Если это уже сделано, сделайте цель активной и добавьте в не файлы.

Цель libtool в подпроекте может быть связана с главным исполняемым файлом (binary-target): щлкните правой кнопкой мыши на нм и выберите Параметры. На вкладке "Библиотеки" выберите соответствующую вспомогательную библиотеку (файл .la)


Что делать, если automake и др. завершается со странными ошибками?

Откройте консоль в корневой папку проекта и выполните: 

libtoolize --copy --force

Это должно решить проблему конфликта версий autotools KDevelop и системной.


У меня появляется сообщение "Wrong autoconf version" или похожее. Что делать?

Есть несколько ошибок о старых версиях autoconf и т.п. Запустите autoreconf в папке, где находятся файлы configure.in, для которых появляются ошибки. Это команда обновит информацию о системных сборочных файлах GNU. Смотрите man autoreconf для подробной информации.

Как можно писать программы для Qt4 в KDevelop-3.4.x?

  • Откройте Проект -> Параметры проекта -> Поддержка C++. Перейдите на вкладку Qt и отметьте флажок Qt4. Затем укажите главную папку установки Qt4, например: 
/usr/share/qt4 (на Debian Unstable)
  • Если для сборки проекта используется qmake, переключить версию библиотеки Qt можно в менеджере qmake. Откройте его и в контекстном меню какого-либо каталога выюерите Параметры. На странице Конфигурация будут показаны разные библиотеки Qt4, которые вам и нужно выбрать

Как можно писать программы для Qt4 в KDevelop-3.3.x?

  • Если вы используете qmake, KDevelop запускает qmake из переменной окружения PATH, так что qmake из qt4 должен быть первым в PATH (http://bugs.kde.org/show_bug.cgi?id=119107). Поэтому для запуска kdevelop используйте сценарий: 
#!/bin/sh
export QTDIR=/devel/kde/src/4/qt-copy
export PATH=$QTDIR/bin:$PATH
exec kdevelop $*
  • Вторая сложность состоит в том, что встроенный Qt designer является копией designer из Qt3, тогда как для .ui файлов нужно использовать Qt4 designer:
    • Откройте ~/.kde/share/config/kdeveloprc, в секции [General Параметры] (не [General]!), добавьте строку: 
Embed KDevDesigner=false
    • Создайте ~/.kde/share/applnk/.hidden/designer.desktop со следующим содержанием (измените путь к Qt4 в соответствии с вашим дистрибутивом) 
[Desktop Entry]
Type=Application
Exec=/devel/kde/src/4/qt-copy/bin/designer
Name=Qt Designer
GenericName=Interface Designer
MimeType=application/x-designer
Icon=designer

Полный путь к Qt дизайнеру необходим там, даже если он корректно указан в PATH в KDevelop. Это связано с тем, что KDevelop запускает Qt designer не напрямую, а через kdeinit.

  • Используйте "Hello World" для создания нового проекта вместо "Application", иначе ваш первый Qt4 проект будет проектом по портированию Qt3->Qt4 :)


Как можно использовать SCons в проекте?

SCons - программа сборки приложений. KDevelop (на момент версии 3.2.2) не поддерживает SCons напрямую. Однако, это легко обойти, создав Makefile-заглушки и создав проект "Custom Makefiles". Вот пример такого Makefile (учтите, что строки после задания имени цели должны начинаться с табуляции, а не пробелов): 

project_name:
        scons
clean:
        scons --clean

После этого заработают действия Собрать проект и Очистить проект


Библиотеки

Как создать динамическую библиотеку (.so)?

  • Добавьте цель в проект
    • Первичная: program
    • Префикс: bin
    • Имя файла: libxxx
    • Параметры компоновщика->Другие: -shared -Wl,-soname,libxxx.so.1 -o libxxx.so.1.0.1
    Однако, на некоторых системах это может привести к следующей ошибке при компиляции:
    -soname command not recognized.
    В этом случае укажите только -shared -o libxxx.so
  • Добавьте cpp/h файлы для данной цели
  • Задайте такие флаги компилятора для c++: -fPIC

Также есть способ, при котором задействуется libtool:

  • Добавьте подпроект "libxxx"
  • Добавьте цель
  • Выберите "Библиотека (libtool)"
  • Выберите "lib".
  • Впишите имя библиотеки "xxx"
  • Добавьте файлы
  • Соберите проект (командой 'Собрать проект' или F8)

Теперь у вас должна быть новая библиотека с именем libxxx.so.0.0.0


Как создать статическую библиотеку (.a)?

  • Добавьте цель в проект
    • Первичная: program
    • Префикс: noinst
    • Имя файла: имя_библиотеки
  • Добавьте файлы cpp/h в цель


Как создать библиотеку .la ("архив libtool")?

  • Добавьте цели в проект
    • primary: Библиотека Libtool
    • prefix: noinst
    • file name: имя_библиотеки
  • Добавьте файлы cpp/h для цели


Как добавить в проект внешний путь к заголовочным файлам?

Если проект использует GNU Autotools (Autoconf/Automake), вы наверняка используете KDevelop Automake manager (см. FAQ#Где находится менеджер Automake в KDevelop?). Чтобы добавить путь к заголовкам:

  • выберите подпроект, куда следует добавить путь(в верхней часть AutoMake менеджера)
  • выберите Параметры..., щлкнув правой кнопкой мыши или нажав на значке с гаечным ключом
  • в вкладке Заголовочные файлы добавьте нужный путь во второе окно (Каталоги вне проекта). Введите путь после -I (без промежуточного пробела).

Как связать программу с динамической библиотекой (.so)?

Если ваш проект использует GNU Autotools (Autoconf/Automake), вы наверняка используете KDevelop Automake manager (см. FAQ#Где находится менеджер Automake в KDevelop?). Чтобы связать программу с библиотекой или добавить библиотеку:

  • выберите подпроект, в верхней часть AutoMake менеджера
  • выберите Параметры..., щлкнув правой или левой кнопкой мыши на значке с гаечным ключом
  • На вкладке "Компилятор" добавьте библиотеку во второе поле (Компоновать с библиотеками вне проекта). Введите имя библиотеки без префикса lib и без расширений .a, .so или .la.
    • Например, если вам надо добавить libsomelib.so, введите -lsomelib.
  • Если эта библиотека находится не в стандартном каталоге для системных библиотек, добавьте -L/некоторый/путь в LDFLAGS в Проект->Параметры проекта...->Параметры configure->Параметры компоновщика (LDFLAGS):.

Не забудьте добавить месторасположение заголовочных файлов *.h библиотеки: FAQ#Как добавить в проект внешний путь к заголовочным файлам?

Как связать программу с библиотекой *.la?

Аналогично FAQ#Как связать программу с динамической библиотекой (.so)?

Дополнение: Есть другой, более удобный, способ компоновки mybinary с somestuff.la. Допустим, что libsomestuff.la находится в $(somedir), тогда вместо -lsomestuff -L$(somedir) можно просто добавить $(somedir)/libsomestuff.la. Тогда они будет обработана libtool.

Вот полный процесс (по аналогии с инструкциями выше):

  • Выберите подпроект, в который следует добавить путь (в верхней часть Automake manager)
  • Выберите Параметры..., щлкнув правой или левой кнопкой мыши на значке с гаечным ключом.
  • НА вкладке "Компилятор" замените -l на $(somedir)/libsomestuff.la во втором поле (Компоновать с библиотеками вне проекта).

Это идентично компоновке вспомогательных библиотек в пределах проекта, только при этом отмечается флажок (Компоновать с библиотеками в проекте).

Нюанс: Библиотеки добавляются в mybinary_LDADD в соответствующем Makefile.am. Переменная LDADD может обрабатывать архивы libtool напрямую, читая расположение нужной библиотеки и соответственно добавляя е.

При сборке библиотек libtool и связи их с исполняемым файлом libtool создат сценарии-посредники для них, но gdb пишет "not in executable format: File format not recognized (неисполняемый формат: тип файла не распознан")

Для того чтобы вс заработало:
Проект -> Параметры проекта -> Отладчик -> Отладочная оболочка -> libtool --mode=execute


Автодополнение кода

Как включить автодополнение кода?

Откройте проект C/C++, зайдите в меню Проект->Параметры проекта, выберите вкладку CПоддержка C++, затем вкладку Авто-дополнение кода. Отметьте Автоматическое дополнение кода и другие параметры. Также в Настройка->Настроить KDevelop->Анализ C++ можно подключить и настроить анализатор, работающий в фоновом режиме.

Как уведомить анализатор об объявлении макрокоманды для С/С++?

В настоящее время анализатор KDevelop обрабатывает файлы не по порядку. (Прим. перев.: Очевидно, имеется в виду, что он не отслеживает зависимости между файлами). Поэтому, для гарантирования того, что анализатору будут известны все макрокоманды, они должны помещаться их в отдельный специальный файл, который будет анализироваться для данного проекта всегда первым. Его содержимое задатся в окне Параметры анализатора C++.

Как включить автодополнение для внешних библиотек?

Если у вас открыт С/С++ проект, откройте в Проект->Параметры проекта, перейдите на вкладку Поддержка C++, затем Авто-дополнение кода. Затем нажмите "Добавить...", после чего следуйте указанием утилиты создания хранилища классов. После завершения настройки убедитесь, что только что созданный класс выбран в списке "хранилища классов" на вкладке Авто-дополнение кода.

Автодополнение кода по-прежнему не работает

Попробуйте отключить компонент "Автозамена", он обычно опережает и отменяет действия по авто-дополнению.

Почему хранилище классов Qt4 не работает для всех классов?

Большинство заголовочных файлов Qt4 являются посредниками и содержат в себе что-то вроде '#include "../../src/blah.h", что KDevelop не может обработать. Укажите реальный каталог заголовочных файлов при создании базы данных автодополнения.


Установка KDevelop

Сколько времени занимает компиляция kdevelop?

Это зависит от производительности вашей машины. На P4 3GHz компиляция занимает 40 минут в первый раз. После обновления через CVS, обычно время сборки сокращается. Мы предполагаем, что время сборки может быть от 1 часа до двух после первой загрузки кода приложения.

Используйте gcc-4.0.1 или выше. Это может значительно уменьшить время компиляции. Компиляция с включнной отладкой ("./configure --enable-debug=full" с -g -O0) выполняется быстрее на 45%, а окончательная компиляция ("./configure --enable-debug=no" с -O2) - на 16% по сравнению с предыдущими версиями. Например, на Athlon XP 1600 / 256Mb RAM новая сборка с отладкой занимает 45 минут при использовании gcc-4 и 81 минуту с gcc-3.3.1. Чтобы ещ сильнее ускорить сборку можно использовать самый свежий пакет libtool (из CVS HEAD)

Обычно удобно устанавливать gcc-4 в /usr/local, а gcc-3.x в /usr. Кроме того, KDevelop возможно собрать с gcc-4 не пересобирая остальное ПО (Qt, KDE...).

Замечание! Не используйте gcc-4.0.0, так как он может некорректно компилировать код! gcc-4.0.1 и более поздние версии лишены этой неполадки.


Система выдат ошибку "couldn't create a language plugin (не удалось создать языкового модуля)". Что не так?

Как и для любого другого приложения KDE, следует проверить, распознал ли KDE модули KDevelop. Либо установите их в каталог KDE (например, ./configure --prefix=/opt/kde3), либо добавьте путь для KDevelop в переменную KDEDIRS и запустите "kbuildsycoca". Пример команды в консоли (для случая, когда KDevelop установлен в /home/harry/kdevelop): 

export KDEDIRS=/home/harry/kdevelop:$KDEDIRS && kbuildsycoca



Я получаю сообщение об ошибке "error: qassistantclient.h: No such file or directory". Как его устранить?

В вашем дистрибутиве (SUSE, Mandriva, Fedora, Ubuntu, ...) Qt 3 разделена на несколько пакетов. Установите пакет с заголовочными файлами. Он может называться, например, так:

  • qt3-apps-dev
  • qt3-devel-doc
  • qt3-doc-devel

Лицензии


Можно ли разрабатывать коммерческие приложения, используя KDevelop?

Любой человек может использовать KDevelop по своему усмотрению. Сгенерированный код может распространяться под любыми лицензиями, в генератор даже добавлена возможность вставлять свои копирайты в генерируемые файлы, но лицензией по умолчанию выбирается GPL. Вместо не вы можете добавить свою лицензию. Для редактирования и прочих нужд KDevelop использует другие GPL программы, не делая акцента на том, что использует. Вы можете писать коммерческие/закрытые программы. Используя библиотеки KDE или любые другие свободные библиотеки, нужно следить только за лицензионными соглашениями сторонних библиотек. Для разработки коммерческих приложений с использованием KDE/Qt вам нужна коммерческая лицензия от Trolltech. Библиотеки KDE используют LGPL, что позволяет использовать библиотечные вызовы, компоновать программу с библиотеками KDE. Это относится к библиотекам kdelibs, kdebase (konqueror, kcontrol и т.д.), koffice (для создания приложений koffice). Для дополнительной информации посетите KDevelop Licensing.


Разрешено ли создавать коммерческие модули для KDevelop?

KDevelop поставляется с библиотекой libkdeveloplgpl.so для создания коммерческих модулей. Она предоставляет интерфейсы и утилиты по лицензии LGPL. Если у вас есть коммерческая лицензия от Trolltech, вы можете компоноваться код с libkdeveloplgpl и выпускать программу под любой лицензией.


Прочее

Я разрабатываю приложение, нуждающееся в обработке исключений. Как мне это реализовать?

Отредактируйте файл configure.in(.in) в корневой папке исходных кодов и создайте в нм макрокоманду AC_CHECK_COMPILERS(): 

CXXFLAGS="$CXXFLAGS $USE_EXCEPTIONS"

после этого пересоздайте сценарий "configure", выполнив 

make -f Makefile.dist

в корневой папке исходников и запустите ./configure.


В configure.in.in отсутствует вызов AC_CHECK_COMPILERS(). Куда следует поместить CXXFLAGS="$CXXFLAGS $USE_EXCEPTIONS", упомянутый выше?

Новые версии configure.in.in были упрощены, теперь возможно добавить эту макрокоманду после AM_INIT_AUTOMAKE(<package-name>,<version>).

Как добавить в проект значки?

Сначала добавьте новый подпроект automake с именем pics, затем в его контекстном меню выберите Параметры" -> Префиксы". Добавьте новый префикс, например, mypics и путь $(kde_datadir)/$project/pics. После этого создайте новую цель в подпроекте pics и выберите в качестве первичной цели данные, в качестве префикса - mypics и любое подходящее имя. Затем добавьте значки для новой цели. Как использовать различные компиляторы (системные или кросс-компиляторы)

Компиляторы выбираются для проекта в:

  • Проект->Параметры проекта->Параметры Configure (только для проектов autotools, для других проектов такой вкладки нет)
  • Добавьте новое имя для конфигурации, например, "<имя компилятора>-<платформа>-debug(отладка)"

Вкладка "Общие":

  • задайте ключи configure (например, ключи --host и --build для кросс-компиляции)
  • задайте папку сборки, для избежания проблем используйте то же имя, что и для configure.
  • задайте нужные переменные (LDFLAGS, ENV)

Вкладка "C":

  • задайте имя компилятора C, например, CC, gcc, icc, kcc, ... (в т.ч. и кросс-компиляторы!)
  • укажите подходящие флаги (CFLAGS)

Вкладка "C++":

  • задайте имя компилятора C++, например, cc, g++, ... (в т.ч. и кросс-компиляторы!)
  • укажите подходящие флаги (CXXFLAGS)

Вкладка "Fortran":

  • задайте имя компилятора Fortran, например, g77, g90, gfortran, ... (в т.ч. и кросс-компиляторы!)
  • укажите подходящие флаги (FFLAGS)

Закройте меню опций, выберите конфигурацию сборки в Проект->Конфигурация сборки, затем запустите Automake и др., configure и make.

Например, для того чтобы создать конфигурацию кросс-компилирования для sparc-sun-solaris2.9:

  • Проект -> Параметры проекта -> Параметры configure
  • Добавьте новое имя конфигурации, например, "sparc-debug""
  • Аргументы для сборки: 
--host=i686-pc-linux-gnu -- build=sparc-sun-solaris2.9
  • Папка сборки: 
sparc-debug
  • На вкладке С измените имя компилятора на кросс-компилятор, например, sun-solaris-sparc2.9-gcc
  • На вкладке С++ измените имя компилятора на кросс-компилятор, например, sun-solaris-sparc2.9-g++
  • Automake и др.
  • configure
  • make

Информация по кросс-отладке: Remote debugging using KDevelop 3.0 in comp.os.linux.embedded newsgroup или Cross_debugging. forum также содержит немного информации.

Убедитесь, что исполняемые файлы и библиотеки для кросс компилятора доступны через переменные $PATH and $LD_LIBRARY_PATH!

Бывает необходимо поменять некоторые макрокоманды autoconf/automake в configure.in для полной поддержки компилятора -- почитайте последнее сообщение в этом документе 

forum как заставить gfortran (компилятор GNU F90) работать.


Работа с системой видов (представлениями информации) Clearcase в KDE

Файлы Clearcase обычно находятся в '/usr/vobs/...', исполняемые файлы могут содержать на них ссылки. Если ваш проект KDevelop использует другой путь (например, '/view/[имя снимка]/usr/vobs'), возможны некоторые проблемы с двойным открытием файлов, создаёт проблемы с точками останова и редактированием.

Если kdevelop запускается из консоли в представлении clearcase, но KDE - нет, '/usr/vobs/...' всё-равно будет недоступно. Это можно решить таким образом:

cp /etc/X11/xinit/xinitrc ~/.xinitrc
замените предпоследнюю строчку с
exec $WINDOWMANAGER
на
/opt/rational/clearcase/rhat_x86/bin/cleartool setview -exec $WINDOWMANAGER ViewName

где ViewName - имя вида, который используется.

К сожалению, это может создать проблемы для не-KDE приложений (например, firefox) и привести к сбою всей системы. В таком случае попробуйте использовать другой графический менеджер, такой как Gnome.
